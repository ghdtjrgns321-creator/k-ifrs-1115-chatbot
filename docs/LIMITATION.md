# Formed 답변의 의도된 설계 한계

> 이 문서는 UX 3/4단계(Clarify/Force Conclusion)에서 formed(형식화된) 답변의 설계 한계와 그 이유를 기록합니다.

---

## 1. 배경: 환각 방지 5단계 레이어

이 프로젝트의 핵심 차별점은 LLM 환각을 **구조적으로 차단**하는 5단계 방어 레이어입니다.

| # | 방어 레이어 | 메커니즘 |
|---|------------|---------|
| 1 | Domain 큐레이션 | MASTER_DECISION_TREES — 체크리스트 + 결론가이드 + 선례 |
| 2 | PydanticAI Structured Output | `selected_branches` + `cited_paragraphs` 구조 강제 |
| 3 | result_validator | 빈 인용/빈 분기 → `ModelRetry` 자동 재호출 |
| 4 | reasoning_guard | 결론가이드에 없는 분기 생성 금지 |
| 5 | Precedents 직접 주입 | 큐레이션된 IE/QNA/감리사례 원문을 context에 주입 |

**결과**: LLM이 근거 없는 답변을 생성할 수 없는 구조 → 2종오류(실재 오류 누락) 방지

## 2. Formed 답변이 대응 가능/불가능한 유형

### 대응 가능

| 유형 | 예시 | 처리 경로 |
|------|------|----------|
| 체크리스트 Q&A | "네, 고객이 개별적으로 효익을 얻을 수 있습니다" | clarify → follow_up_questions |
| 추가 조건 제시 | "계약 기간은 3년입니다" | clarify fast-path → 분기 좁히기 |
| 결론 확정 요청 | "결론을 내려주세요" | force conclusion (is_conclusion=True) |
| 개념 질문 | "변동대가란 무엇인가요?" | generate_agent → GenerateOutput |

### 대응 불가능 (한계)

| 유형 | 예시 | 불가능한 이유 |
|------|------|-------------|
| 사례 요청 | "관련 사례 없어요?" | clarify_agent는 체크리스트 Q&A 전용. 사례 검색은 성격이 다른 요청(정보 탐색 vs 판단 확인) |
| 자유형 질문 | "다른 회사는 어떻게 처리해요?" | ClarifyOutput 스키마에 자유 텍스트 필드 없음 |
| 범위 외 교차 질문 | "리스(1116호)와 결합하면?" | MASTER_DECISION_TREES가 K-IFRS 1115호만 커버 |
| 계산 결과 재검증 | "이 계산이 맞나요?" | fast-path에서 retrieve 스킵 → 새로운 문서 검색 불가 |

## 3. 검토한 대안과 채택하지 않은 이유

### 대안 A: UI 수준 해결 (사례 버튼 추가)

- **방법**: AI 답변 패널에 "관련 사례 보기" expander 추가
- `pinpoint_panel.py`의 사례 렌더링 함수를 재사용하여 IE/QNA/감리사례 표시
- **장점**: LLM 호출 0회, formed 답변 설계와 무충돌
- **불채택 이유**: 구현은 가능하나, 이미 좌측 evidence 패널에 동일 사례가 표시됨 → 정보 중복. UX 2단계(토픽 브라우즈)에서도 4탭으로 사례 열람 가능. 추가 UI 없이도 사용자가 사례에 접근할 수 있는 경로가 이미 존재함

### 대안 B: 파이프라인 수준 해결 (사례 요청 감지 + 별도 경로)

- **방법**: fast-path에서 "사례" 키워드 감지 시 retrieve 재실행
- **장점**: 사용자 의도에 정확히 대응
- **불채택 이유**: fast-path의 성능 이점(analyze/retrieve/rerank 스킵) 상실. 키워드 감지 자체가 새로운 분기 복잡도 추가. formed 답변의 structured output 보장 해제 필요

### 대안 C: 프롬프트 수준 해결 (clarify_agent 확장)

- **방법**: clarify_agent에 "TYPE 3: 사례 제시 모드" 추가
- **장점**: 기존 파이프라인 구조 유지
- **불채택 이유**: ClarifyOutput 스키마 변경 필요 → result_validator + reasoning_guard 전면 수정 → 환각 방지 레이어 2,3,4 약화 위험. "사례를 자유롭게 인용하라"는 지시는 환각 방지 설계 철학과 정면 충돌

### 대안 D: Graceful 거절

- **방법**: "사례는 좌측 근거 패널에서 확인해주세요" 안내 메시지 반환
- **장점**: 구현 간단, 사용자 혼란 최소화
- **불채택 이유**: 별도 구현할 필요 없이, 현재 설계에서 이미 evidence 패널이 이 역할을 수행

## 4. 결론: 의도된 설계 한계

이 한계는 **버그가 아닌 의도된 trade-off**입니다.

> 이 설계는 **규칙 기반 전문가 시스템**에 가깝습니다.
> — PROJECT_OVERVIEW.md

### 설계 원칙

- **"근거 선행, AI 후행"**: 사용자가 먼저 팩트(근거 문서)를 확인한 뒤 AI에게 질문하는 구조
- **LLM 역할 = "전달자"**: 큐레이션된 결론가이드를 그대로 전달, 자유 추론 제한
- formed 답변의 제약 = 환각 방지의 핵심 메커니즘

### 사용자가 사례를 찾는 기존 경로

1. **UX 2단계 (토픽 브라우즈)**: 4탭에서 IE/QNA/감리사례 직접 열람
2. **UX 3단계 (evidence 패널)**: 좌측에 카테고리별 아코디언으로 관련 문서 표시
3. **AI 답변의 cited_paragraphs**: AI가 인용한 근거 문단 목록
4. **감리사례 넛지**: format 노드가 유사 감리사례를 자동 매칭하여 경고 표시

---

## 5. 추후 개선 방향

현재 프로젝트 범위에서는 위 한계를 유지하되, 향후 다음 방향을 검토할 수 있습니다:

- **듀얼 모드 (V2)**: formed 답변 완료 후, "자유 질문 모드로 전환" 버튼 → 별도 generate_agent로 자유형 답변 생성 (단, 환각 방지 레이어 약화 감수)
- **멀티턴 컨텍스트 확장**: clarify fast-path에서도 부분적 retrieve 허용 (성능과 품질 균형점 탐색)
