# 모델 비교 테스트 최종 보고서

## 1. 테스트 목적

K-IFRS 1115(수익 인식) 회계 챗봇의 **LLM 모델 선정**을 위해, 7개 모델을 대상으로
**회계 추론 품질, 산술 정확도, 포맷 준수율, 비용**을 정량 비교하였다.

테스트를 통해 다음 질문에 답하고자 했다:
- 회계 판단 질문과 산술 계산 질문에 **단일 모델이 최적인가, 듀얼트랙이 필요한가?**
- 리트리버 커버리지가 생성 품질에 얼마나 영향을 미치는가?
- 프로덕션 파이프라인에서 calc 라우팅은 어떻게 설계해야 하는가?

## 2. 테스트 설계

### 평가 대상 모델 (7개 → 6개 → 듀얼트랙)

| 모델 | 유형 | 비고 |
|------|------|------|
| Gemini 3 Flash (low/medium/high) | Reasoning | thinking_level 3단계 |
| gpt-4.1-mini | Non-reasoning | 최저 비용 |
| gpt-5-mini (low/medium) | Reasoning | OpenAI 최신 |
| o4-mini (medium) | Reasoning | 1차에서 탈락 |
| Solar Mini | Non-reasoning | 1차에서 탈락 |

### 골든 질문 (T1~T5)

| ID | 토픽 | 유형 | 난이도 |
|----|------|------|--------|
| T1 | 본인/대리인 — 위탁판매 | 상황 판단 | 최고 |
| T2 | 변동대가 이론 | 개념 설명 | 낮음 |
| T3 | 변동대가 — 볼륨디스카운트 | 상황 적용 | 중간 |
| T4 | 계약식별 — 가공매출/회수가능성 | 상황 판단 | 중간 |
| T5 | 진행률 계산 — 투입법 | 산술 계산 | 중간 |

### 평가 메트릭

- **채점 (0~1)**: LLM 기반 자동 평가 (quality_evaluator.py)
- **환각 Pass**: 근거 없는 회계 처리 생성 여부
- **포맷 Pass**: `**[결론]**` + `**[논리적 근거]**` 구조 준수
- **검색 적중률**: retrieval_targets 대비 실제 검색된 문서 비율

## 4. 핵심 개선사항 상세

### 개선 1: `_COT_PREFIX` 제거 → 산술 정확도 회복 (ADR-10)

- **문제**: 3차에서 user msg 앞에 prepend된 `_COT_PREFIX`(포맷 지시)가 시스템 프롬프트(`CALC_CLARIFY_SYSTEM`)와 중복 → gpt-4.1-mini 산술 정확도 0.845→0.67 급락
- **원인 분석**: Non-reasoning 모델은 동일 지시가 system + user 양쪽에 있으면 포맷 규칙에 attention이 분산되어 산술 로직 집중도 하락
- **해결**: `_COT_PREFIX` 변수 정의 + 3개 사용처 완전 제거
- **효과**: T5 산술 정답률 1/3 → **3/3**, 점수 0.67→**0.83**

```python
# Before: user msg에 포맷 지시 prepend
user_msg = _COT_PREFIX + user_msg  # 시스템 프롬프트와 중복!

# After: 시스템 프롬프트에만 포맷 지시 유지
user_msg = user_msg  # _COT_PREFIX 완전 제거
```

### 개선 2: calc 경로에서 topic_knowledge 스킵 (ADR-10)

- **문제**: topic_knowledge(~2000자)가 calc 경로에서 불필요 토큰 노이즈 → 산술 집중도 분산
- **해결**: `use_calc=True`이면 topic_knowledge 미주입

```python
# generate.py L307~312
# Why: topic_knowledge가 gpt-4.1-mini의 산술 집중도를 분산시킴
if not use_calc:
    topic_knowledge = _format_topic_knowledge(state.get("matched_topics", []))
    if topic_knowledge:
        context_str = f"[참고 지식]\n{topic_knowledge}\n\n---\n\n{context_str}"
```

### 개선 3: calc 라우팅 v1(regex) → v2(LLM 판단) (ADR-2)

- **문제**: v1은 regex 3조건 AND(`_CALC_COMMAND` + `_AMOUNT_PATTERN` + `has_formula`)로, LLM search_keywords 비결정성에 의해 1/3만 calc 진입
- **해결**: `AnalyzeResult.needs_calculation: bool` — analyze_agent(gpt-4.1-mini)가 LLM으로 판단
- **효과**: 42/42 정확도 100%, T3 같은 회계 판단 질문은 올바르게 Gemini로 라우팅

### 개선 4: IE bypass (ADR-21)

- **문제**: IE 적용사례 fetch 후 상한 3개 제한 → 관련 사례 청크 누락
- **해결**: calc 경로에서만 IE 제외, 일반/상황 질문에서는 상한 제거

### 개선 5: 핀포인트 3중 Gap 수정 → 검색 적중률 36%→76% (ADR-22)

- **문제**: `_parse_doc_ids_from_text()`의 3가지 파싱 결함

| Gap | 원인 | 수정 |
|-----|------|------|
| 괄호 미파싱 | 섹션 2 checklist가 `(QNA-xxx)` 형태 사용 | 괄호 + 대괄호 양쪽 파싱 |
| 축약 ID 불일치 | `FSS-2022-...` vs DB의 `FSS-CASE-2022-...` | 자동 정규화 |
| critical_factors 미수집 | 섹션 7 텍스트 스킵 | `checklist` + `critical_factors` 모두 수집 |

- **효과**: 검색 적중률 36%→**76%** (+40%p)

### 최종 프로덕션 라우팅 규칙

```

  → needs_calculation = True?
```

> `calculation_formula`는 라우팅 조건이 아니라, calc 경로의 컨텍스트에 공식 텍스트를 주입하는 용도이다.

## 6. 4차 이후 추가 개선 (ADR-23~30)

모델 비교 테스트 이후 **품질 테스트(골든 53건)** 과정에서 발견된 추가 개선사항.

### 6-1. 토픽 매칭 고도화

| ADR | 개선 | 효과 |
|-----|------|------|
| **ADR-19** | `topic_hints` — analyze_agent가 LLM으로 관련 토픽 힌트 제공 | "재판매" 같은 비정형 질문에서 "본인/대리인" 토픽 매칭 성공 |
| **ADR-23** | 토픽 임베딩 코사인 유사도 — 3중 매칭 (keyword + topic_hints + embedding) | 일상 언어 질문의 토픽 매칭 안정화 |
| **ADR-30** | 토픽 매칭 진단 기준 top-1 → top-3 | 거짓 양성 18→6건 감소 |

### 6-2. thinking_level 최적화

| ADR | 개선 | 효과 |
|-----|------|------|
| **ADR-27** | thinking: high → **medium** | 과도한 추론의 비결정성 제거, 꼬리질문 일관성 60%→100% |
| **ADR-28** | 후속 턴 thinking: medium → **low** + fast-path 활성화 | M02 3턴 136초→86~104초 (30% 개선) |

### 6-3. 멀티턴 품질 안전장치

| ADR | 개선 | 효과 |
|-----|------|------|
| **ADR-17** | `concluded` 플래그 + `critical_factors` 양방향 분기 가드 | 결론 후 불필요 추가 질문 방지, TYPE 2 단정 방지 |
| **ADR-18** | ANALYZE_PROMPT 범위 밖 키워드 가드 | 타 기준서(1116호 등) 용어 혼동 방어 |

### 6-4. UI/검색 품질

| ADR | 개선 | 효과 |
|-----|------|------|
| **ADR-24** | pinpoint 미인용 문서를 "참고하면 좋은 추가 문서"로 표시 | fetch된 10~15건이 UI에서 누락되지 않음 |
| **ADR-25** | source 문자열 + ID 접두어 중앙화 | 10+파일의 하드코딩 제거, 그룹핑 불일치 방지 |
| **ADR-26** | AI 답변 좌측 패널에 IE/QNA/감리 3계층 표시 | 근거 문서 가시성 대폭 향상 |

## 7. 남은 과제 해결 현황

| 과제 | 상태 | 해결 내용 |
|------|------|----------|
| T1 본인/대리인 (0.62) | **해결 ✅** | trigger_keywords 20개 확장 + ADR-17(양방향 안전장치) + ADR-19(topic_hints). 골든 테스트 S01/S06/M01 모두 PASS |
| T3 검색 적중 (1/5) | **해결 ✅** | `_parse_doc_ids_from_text()`에 콤마 구분 파싱 구현 (정규식+split 2단계). "문단 59, 87" → 개별 조회 동작 확인 |
| T2 검색 적중 (3/5) | **부분 해결 ⚠️** | 아래 상세 참조 |

### T2 검색 적중 — 부분 해결 판정 근거

**개선된 부분**:
- ADR-23(3중 매칭: keyword + topic_hints + 임베딩 유사도)으로 토픽 매칭 정확도 60%→100% 안정화
- 골든 테스트 전체 88.7%(47/53) 통과, 거래상황(S) 90%, 계산(C) 100%, 개념(K) 100%

**미해결로 남긴 이유 — 리스크 대비 효용 부족**:
- 미해결 6건(S05, S12, N08, X03 등)은 decision_trees에 해당 토픽 자체가 없거나, 다중 토픽 교차 질문이라 **구조적 확장**(새 토픽 추가, top-3 병렬 체크리스트)이 필요
- 이 구조 변경은 기존 47건의 회귀 위험을 수반하며, 6건 개선 효용 대비 회귀 테스트 비용이 과도
- T2 원래 문제("concept 질문의 retrieval_targets 설계 한계")는 평가 설계 자체의 한계로, 실제 답변 품질에는 영향 미미 — 개념 질문(K) 100% 통과가 이를 증명

### Gemini 산술 오류 — 우회 해결 판정 근거

**우회 전략과 효과**:
- `needs_calculation=True` → gpt-4.1-mini로 라우팅 (ADR-2 v2)
- calc 경로에서 topic_knowledge 스킵 + IE pinpoint 제외 → 산술 집중도 확보 (ADR-10, 21)
- 골든 테스트 계산 케이스(C01~C03) **100% 정답** 달성

**근본 해결을 하지 않은 이유 — 프로젝트 범위 밖**:
- Gemini의 산술 오류는 **모델 자체의 한계**(분모에서 낭비원가 미차감 등)로, 프롬프트 엔지니어링이나 컨텍스트 최적화로는 해결 불가
- Gemini 측의 모델 업데이트 외에 해결 수단이 없으며, 이는 본 프로젝트의 통제 범위를 벗어남
- 듀얼트랙 라우팅이 실질적으로 이 문제를 완전히 차단하고 있어, 추가 투자의 기대 효용이 0에 가까움 — 계산 질문은 gpt-4.1-mini가 100% 처리하므로 Gemini가 산술을 수행할 경로 자체가 존재하지 않음
