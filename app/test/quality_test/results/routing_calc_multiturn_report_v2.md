## 5. 근본 원인 분석 + 다음 개선 방향

### 문제 1: B1/B2 calc 라우팅 2/3만 성공 (needs_calculation 비결정성)

**원인**: analyze_agent가 동일 질문에 대해 `needs_calculation`을 가끔 False로 반환. 단독 테스트(14/14)와 달리, 파이프라인 내에서는 대화 맥락("최신 대화 기록 및 질문: human: ...")이 포함되어 프롬프트 환경이 다름.

**가능한 개선**:
- (A) LLM 판단 + regex fallback 하이브리드: LLM이 False여도 regex(`구해줘`+금액3개)면 True로 override
- (B) ANALYZE_PROMPT의 `[needs_calculation]` 예시를 더 강화

### 문제 2: C1 양방향 안전장치 과잉 (v1 Pass → v2 Partial)

**원인**: `_validate_clarify()`가 `critical_factors`의 키워드를 `checked_items` + `provided_info`에서 매칭하지 못함. 사용자가 "재고위험도 저희가 부담"이라고 했지만 factor는 "재고위험 부담 여부"로 되어있어 `_factor_in_text()` 매칭 미스.

**가능한 개선**:
- `_factor_in_text()` 검사 범위를 user message 본문까지 확장
- 또는 critical_factors 키워드를 더 짧고 유연하게 조정

### 문제 3: C2 T3 follow_up 미제거 (concluded 미작동)

**원인**: `concluded=True`가 세션에 저장되고 [결론 확인 모드] 프롬프트가 주입되지만, Gemini thinking이 프롬프트 지시를 무시하고 follow_up_questions를 생성.

**가능한 개선**:
- (A) `_validate_clarify()`에 concluded 체크 추가: concluded=True면 follow_up 비어있지 않으면 ModelRetry
- (B) post-processing에서 concluded=True이면 follow_up 강제 빈 배열로 치환

### 문제 4: C3 범위 밖 감지 실패 (0/3)

**원인**: "증분차입이자율"은 LLM이 1115호 "금융요소"와 의미적으로 연결하여 routing="IN" 판정. 프롬프트 수준 키워드 가드로는 LLM의 의미 추론을 override 불가.

**가능한 개선**:
- (A) ANALYZE_PROMPT에 명시적 예시 추가: `"리스 부채 할인율 산정 시 증분차입이자율 → OUT"`
- (B) analyze 호출 전 regex 사전 필터: "증분차입이자율|사용권자산|리스부채" 감지 시 강제 OUT
