# 데이터 파이프라인 — 수집에서 검색까지

> AI가 올바른 답변을 하려면, 올바른 데이터가 먼저 준비되어야 한다.
> 이 문서는 6종의 회계 데이터를 수집하고, AI가 검색 가능한 형태로 가공하는 전처리 파이프라인을 설명한다.

---

## 1. 왜 자체 데이터 파이프라인이 필요한가

검증된 도메인 데이터만을 AI에 주입하여 확정적 근거를 제시한다. 이 파이프라인은 **"AI의 DB를 구축하는 과정"**이다.

범용 AI(ChatGPT)는 학습 시점에 고정된 데이터만 사용하므로, 새로 나온 질의회신이나 최근 감리지적사례를 알 수 없다. 이 시스템은 **6종의 공식 소스를 직접 수집·가공**하여 DB에 저장하고, 검색 시 원문을 제공한다. 데이터 소스가 업데이트되면 해당 단계만 재실행하여 즉시 반영할 수 있다.

---

## 2. 데이터 소스 총괄

총 **6종, 약 1,575건**의 데이터를 수집·가공한다.

| 데이터 유형 | 건수 | 출처 | 수집 방법 | 용도 |
|------------|------|------|----------|------|
| **기준서 본문·적용지침·BC** | ~1,300 청크 | kifrs.com API | REST API 자동 크롤링 | 기준서 원문 검색·인용 |
| **적용사례 (IE)** | ~150 문단 | kifrs.com API | 본문과 동일 API | 실무 적용 예시 |
| **질의회신 (QNA)** | 101건 | kifrs.com (4개 게시판) | 게시판별 타겟 크롤링 | 실무 해석 선례 |
| **감리지적사례** | 18건 | FSS·KICPA | 수동 수집 + 구조화 | 규제 리스크 경고 |
| **KAI 교육자료** | 5건 | 한국회계기준원 | 수동 수집 + 구조화 | AI 답변 품질 보강 |
| **토픽 큐레이션** | 30개 토픽(Decision Trees 26개 + 세부 분할 4개) | AI 교차검증(Gemini·Claude·NotebookLM)으로 작성 | 텍스트 파싱 | 체크리스트 + 결론 가이드 |

### 각 소스가 필요한 이유

- **기준서 본문**: 모든 판단의 출발점. AI가 인용할 수 있는 1차 근거
- **적용지침(B문단)**: 본문의 원칙을 실무에 적용하는 구체적 지침. 감사인이 가장 자주 참조하는 영역
- **적용사례(IE)**: "이런 거래는 이렇게 처리한다"는 공식 예시. 유사 거래 판단의 기준점
- **질의회신(QNA)**: 회계기준원이 구체적 사례에 대해 해석을 내린 **사실상의 판례**
- **감리지적사례**: 금융감독원이 실제로 지적한 사례. 비슷한 상황에서 **같은 실수를 방지**하는 레드플래그
- **KAI 교육자료**: 한국회계기준원이 발행한 실무 교육 콘텐츠. AI 답변의 맥락 보강

### QNA 4개 게시판의 가중치

질의회신은 출처의 권위에 따라 다른 가중치를 부여한다:

| 게시판 | 가중치 | 이유 |
|--------|--------|------|
| IFRS 해석위원회 | 1.15 | 국제 기준 해석 기관 |
| 금융감독원 | 1.15 | 국내 최고 감독 기관 |
| 회계기준원 정규질의 | 1.10 | 공식 질의회신 |
| 신속처리질의(SSI) | 1.05 | 약식 회신, 참고 수준 |

## 3. 파이프라인 전체 흐름

14개 스크립트를 3개 Phase로 나누어 실행한다:

```
Phase 1: 수집 (Crawling)
─────────────────────────────────────────────────────────
  [01] 웹 구조 검사 → [02] 기준서 본문 크롤링
                       [05] QNA 4개 게시판 크롤링

Phase 2: 가공 (Processing)
─────────────────────────────────────────────────────────
  [03] HTML → 마크다운 변환 + 가중치 청킹
  [04] 본문 임베딩 → MongoDB 적재
  [06] QNA 3단 분할(Q/A/S) + 임베딩
  [07] 감리사례 3단 분할 + 임베딩
  [08] KAI 교육자료 임베딩 + LLM 제목 자동 생성
  [09] 검색어 확장 매핑 생성

Phase 3: 큐레이션 (Curation)
─────────────────────────────────────────────────────────
  [10] 토픽 큐레이션 텍스트 → JSON 파싱
  [11] 외부 테이블 보정
  [12] QNA·감리사례 서머리 임베딩
  [13] 토픽 레벨 임베딩
  [99] 교차 검증 (전수 품질 감사)
```

> **왜 이렇게 단계를 나누었는가?** 각 단계가 독립 실행 가능하므로, 데이터 소스 하나가 업데이트되면 해당 단계만 재실행할 수 있다. 예: 질의회신이 추가되면 [05]→[06]→[12]만 실행하면 된다.

## 4. 청킹 전략: 어떻게 잘라야 잘 찾을까

> **청킹(Chunking)이란?**
> 긴 문서를 AI가 검색할 수 있는 작은 조각(chunk)으로 나누는 작업이다.
> 통째로 한 파일에 넣으면 필요한 부분을 찾기 어렵고, 너무 잘게 쪼개면 맥락을 잃는다.

### 4-1. 기준서 본문: 가중치 청킹

기준서 본문은 **문단 단위**로 분할한다. K-IFRS 1115호 자체가 문단 번호 체계를 갖고 있으므로, 이를 그대로 활용한다.

**3단계 변환 과정:**

```
① HTML 원본 → ② 마크다운 변환 → ③ 메타데이터 부착

① kifrs.com에서 가져온 HTML
   <p class="para-inner-number">⑴ 재화나 용역에 대한...</p>

② 정제된 마크다운
   ⑴ 재화나 용역에 대한...

③ 메타데이터
   chunk_id: "1115-B34A"
   category: "적용지침B"
   weight_score: 1.3
   hierarchy: "적용지침 > 본인인지 대리인인지의 판단"
```

**계층 경로(hierarchy)**는 각 청크에 "이 문단이 어디에 속하는지"를 기록한다. AI가 검색할 때 **"적용지침 > 본인 vs 대리인 판단"** 이라는 맥락이 함께 제공되어, 단순히 본문만 보는 것보다 정확한 결과를 얻는다.

**카테고리별 가중치:**

| 카테고리 | 가중치 | 이유 |
|---------|--------|------|
| **본문** | 1.3 | 원칙 수준의 규정 — 가장 직접적인 근거 |
| **적용지침 (B문단)** | 1.3 | 실무 판단 기준으로 본문과 동급 |
| **적용사례 (IE)** | 1.1 | 참고용 예시 |
| **결론도출근거 (BC)** | 0.8 | 제정 배경 이해용 — 실무 직접 적용은 낮음 |
| **용어정의·시행일** | 0.8 | 보조적 정보 — 기본값 이하로 하향 |

> **왜 가중치를 부여하는가?** 기준서 본문을 직접 읽어본 결과, 본문과 적용지침(B문단)이 실무 판단에 가장 직접적인 근거를 제공한다고 판단했다. 이 둘은 동급(1.3)으로 설정하고, 결론도출근거(BC)·용어정의·시행일은 배경 정보에 가까워 기본값(1.0) 이하인 0.8로 하향했다.

### 4-2. QNA·감리사례: 3단 분할 (Q/A/S)

질의회신과 감리사례는 기준서 본문과 다른 구조를 가진다. 하나의 질의회신 안에 **질문(Q)·회신(A)·관련기준(S)**이 모두 담겨 있으므로, 이를 분리하여 저장한다.

```
하나의 질의회신을 3조각으로 분리:

  [Q] 질의 내용 — "이런 거래는 어떻게 처리하나요?"
  [A] 회신 내용 — "이렇게 처리합니다. 근거는 문단 XX입니다."
  [S] 관련 기준 — 참고한 기준서 조항, 부록 목록
```

> **왜 분리하는가?** 사용자가 "변동대가 산정 방법"을 검색하면, Q(질의)와 A(회신) 모두 검색에 걸릴 수 있다. 그런데 Q는 "질문 텍스트"이므로 사용자 질문과 유사도가 높지만, 실제로 AI 답변에 필요한 것은 A(회신)이다. 분리하지 않으면 Q의 유사도가 높아서 정작 A를 밀어낼 수 있다.
>
> 이를 방지하기 위해 Q에는 가중치 **x0.80**, A에는 **x1.0**을 부여하여 회신이 우선 노출되도록 한다.

### 4-3. Parent Document Retrieval (PDR)

> **PDR이란?**
> 작은 조각(child)으로 정확히 검색한 뒤, AI에게는 원문 전체(parent)를 제공하는 패턴이다.
> 목차(child)로 원하는 장을 찾고, 실제로는 그 장 전체(parent)를 읽는 것과 같다.

```
검색 시점:
  사용자 질문 → Child 청크(Q/A/S 조각)에서 검색 → 정확한 매칭

AI 답변 시점:
  매칭된 Child의 Parent ID 확인 → Parent 컬렉션에서 원문 전체 조회
  → AI에게 질의회신 전문을 제공 (질문·회신·관련기준 모두 포함)
```

> **왜 이렇게 하는가?** 작은 조각은 **검색 정확도**가 높지만 맥락이 부족하다. 원문 전체는 맥락은 풍부하지만 검색 시 노이즈가 많다. PDR은 **검색의 정확도**와 **답변의 맥락**을 동시에 확보하는 전략이다.

## 5. 임베딩 전략: 의미를 숫자로 변환

> **임베딩(Embedding)이란?**
> 텍스트의 의미를 수천 개의 숫자(벡터)로 변환하는 기술이다. 의미가 비슷한 문장은 숫자 패턴도 비슷해지므로, "변동대가 산정"과 "볼륨 디스카운트 계산"이 유사하다는 것을 AI가 인식할 수 있다.
>

### 모델 선택: Upstage Solar Embedding

한국어에 특화된 Upstage Solar Embedding을 사용한다. 이 모델의 가장 큰 특징은 **passage(저장)와 query(검색) 모델이 분리**되어 있다는 점이다.

| 용도 | 모델명 | 사용 시점 |
|------|--------|----------|
| **저장** | solar-embedding-1-large-**passage** | 문서를 DB에 넣을 때 |
| **검색** | solar-embedding-1-large-**query** | 사용자 질문을 검색할 때 |

> **왜 분리하는가?** 문서(passage)와 질문(query)은 성격이 다르다. "변동대가의 추정"(문서)과 "변동대가를 어떻게 산정하나요?"(질문)은 같은 주제이지만 표현 방식이 다르다. 각각에 최적화된 모델을 쓰면 검색 품질이 향상된다. **혼용하면 검색 품질이 급락한다** — 프로젝트 초기에 이를 발견하고 엄격히 분리했다.

### 문맥 접두사 전략

각 청크를 임베딩할 때, 앞에 문맥 정보를 붙인다:

```
[문맥: 적용지침 > 본인인지 대리인인지의 판단 > 회신]
재화나 용역이 고객에게 이전되기 전에 기업이 그 재화나 용역을 통제하는지...
```

이렇게 하면 같은 텍스트라도 **"어디에 속한 내용인지"**가 숫자 패턴에 반영되어, 검색 시 더 정확한 결과를 얻는다.

### 제약 조건

- **3,000자 제한**: Upstage 모델의 토큰 한도(4,000) ≈ 한국어 3,000자
- **배치 100건**: API 과부하 방지를 위해 100건 단위로 묶어 처리

## 6. 저장소 구조: MongoDB Atlas

> **MongoDB Atlas란?**
> 클라우드 기반 데이터베이스로, 일반 데이터 저장과 벡터 검색을 모두 지원한다. 별도의 벡터 전용 DB를 쓰지 않고 **하나의 DB에서 텍스트·메타데이터·벡터를 통합 관리**할 수 있어 운영이 단순하다.

### 4개 컬렉션 구조

| 컬렉션 | 용도 | 주요 내용 | 규모 |
|--------|------|----------|------|
| `k-ifrs-1115-chatbot` | **메인** (child 청크) | 본문 + QNA child + 감리 child + KAI child | ~2,700건 |
| `k-ifrs-1115-qna-parents` | QNA 원본 | 질의회신 전문 (PDR용) | 101건 |
| `k-ifrs-1115-findings-parents` | 감리사례 원본 | 감리지적사례 전문 (PDR용) | ~40건 |
| `k-ifrs-1115-kai-parents` | KAI 원본 | 교육자료 전문 (PDR용) | ~25건 |

### 메인 컬렉션의 문서 구조

각 청크에는 다음 정보가 함께 저장된다:

| 필드 | 설명 | 예시 |
|------|------|------|
| `chunk_id` | 고유 식별자 | "1115-B34A", "QNA-SSI-38674_A" |
| `text` | 청크 본문 | "재화나 용역에 대한..." |
| `embedding` | 4,096차원 벡터 | [0.012, -0.034, ...] |
| `category` | 문서 유형 | "적용지침B", "질의회신" |
| `weight_score` | 검색 가중치 | 1.3 |
| `hierarchy` | 계층 경로 | "적용지침 > 본인 vs 대리인" |
| `parent_id` | 원문 참조 (QNA/감리만) | "QNA-SSI-38674" |
| `chunk_type` | 분할 유형 (QNA/감리만) | "question", "answer", "supplementary" |

## 7. 데이터 품질 관리

### 교차 검증

99-verify-chunks 스크립트가 **전수 품질 감사**를 수행한다:

- MongoDB에 적재된 문서 수 ↔ 원본 JSON 문서 수 일치 확인
- topics.json에 기록된 문단 번호가 실제 DB에 존재하는지 확인
- 임베딩 차원(4,096) 일관성 검증
- 중복 chunk_id 탐지

### 유니코드 정규화 교훈

외부 데이터에는 같은 기호라도 **컴퓨터 내부 코드가 다른 경우**가 있다.

**실제 발생한 문제**: "문단 B34\~B38"의 범위를 파싱할 때, 물결표(`~`) 종류가 눈에는 같아 보이지만 내부 코드가 달라 해당 문단 5개 전체가 검색에서 누락되었다. 전처리 단계에서 3종 물결표를 하나로 통일하여 해결했다.

> **교훈**: 외부 데이터를 다룰 때는 "눈에 보이는 것"을 믿지 말고, 내부 코드 수준에서 검증해야 한다.

## 8. 핵심 의사결정 요약

| ADR | 결정 | 결과 |
|-----|------|------|
| **ADR-5** | 2계층 검색: 큐레이션 핀포인트 + 벡터 하이브리드 | 핵심 문서 누락 0% + 탐색적 검색 겸비 |
| **ADR-8** | PDR (Parent Document Retrieval) | 검색 정확도(child) + AI 맥락(parent) 양립 |
| **ADR-12** | passage/query 임베딩 엄격 분리 | 혼용 시 품질 급락 방지 |
| **ADR-16** | 토픽 파싱 단일 파이프라인 (분리 실행 금지) | 2단계 파이프라인의 동기화 누락 사고 방지 |
| **ADR-22** | 핀포인트 파싱 3중 Gap 수정 | 검색 적중률 **36% → 76%** (+40%p) — 프로젝트 최대 단일 개선 |
