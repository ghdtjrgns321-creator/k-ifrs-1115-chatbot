# Decision Tree 큐레이션 — 이 프로젝트의 핵심

> **일반 RAG(검색 증강 생성)는 "관련 문서를 찾아서 AI에게 넘겨주는" 것까지만 한다.**
> 이 프로젝트는 한 단계 더 나아가, **"AI가 어떤 체크리스트로, 어떤 분기 결론만 선택할 수 있는지"**까지 사전에 작성해 둔다.

---

## 1. 왜 큐레이션이 필요한가

### 일반 RAG의 한계

일반적인 AI 검색 시스템은 질문과 유사한 문서를 찾아 AI에게 전달한다. 하지만 검색만으로는 두 가지가 보장되지 않는다:

| 문제 | 설명 |
|------|------|
| **결론의 구조 부재** | AI가 자유형 텍스트로 답변 → 체계적 판단 불가 |
| **선례·감리사례 누락** | 검색 엔진이 놓치면 끝 |

### 이 시스템의 해법: Decision Tree

K-IFRS 1115호의 주요 판단 단위를 **26개 토픽**으로 나누고, 각 토픽에 대해 **체크리스트 → 결론 분기 → 선례 → 감리 경고 → 계산 공식**까지 사전 작성한다. AI는 이 Decision Tree의 범위 안에서만 답변할 수 있다.

> **핵심 원리**: AI에게 "자유롭게 답변해라"가 아니라 **"이 체크리스트를 확인하고, 이 분기 중 하나를 골라라"**고 강제하는 것.

---

## 2. 토픽 체계: 26개 토픽 설계

K-IFRS 1115호의 구조를 따라 26개 토픽으로 분류했다. 각 토픽은 감사인이 실무에서 마주치는 **하나의 판단 단위**에 대응한다.

### 토픽 목록 (기준서 순서)

| # | 토픽 | 판단 내용 |
|---|------|----------|
| 1 | 계약의 식별 | 5요건 충족 여부 → 1115호 적용 대상인가 |
| 2 | 계약의 결합 | 복수 계약을 하나로 묶을지 판단 |
| 3 | 계약변경 | 변경된 계약을 별도 계약·소급·전진 중 어떻게 처리할지 |
| 4 | 수행의무 식별 | 약속한 재화/용역을 몇 개의 수행의무로 나눌지 |
| 5 | 일련의 구별되는 재화나 용역 | 반복적으로 동일한 용역 → 단일 수행의무로 묶을지 |
| 6 | **변동대가** | 할인·위약금·리베이트 등 변동 가능한 대가 추정 |
| 7 | 유의적인 금융요소 | 할부·선수금의 이자 반영 여부 |
| 8 | 비현금 대가 | 주식·현물 등의 대가 측정 |
| 9 | 고객에게 지급할 대가 | 캐시백·장려금을 수익 차감 vs 비용 처리 |
| 10 | 거래가격 배분 | 여러 수행의무에 대가를 어떻게 나눌지 |
| 11 | 할인액의 배분 | 할인이 특정 수행의무에 귀속되는지 |
| 12 | 변동대가의 배분 | 변동대가가 특정 수행의무에 귀속되는지 |
| 13 | **기간에 걸쳐 vs 한 시점 인식** | 수익 인식 시점·패턴 결정 |
| 14 | 진행률 측정 | 투입법·산출법으로 진행률 계산 |
| 15 | 거래가격의 후속 변동 | 배분 후 대가가 바뀌면 어떻게 재배분할지 |
| 16 | 표시 | 계약자산·계약부채·수취채권 분류 |
| 17 | 계약체결 증분원가 | 영업수수료 등 자산화 vs 즉시 비용화 |
| 18 | 계약이행원가 | 계약 이행을 위해 발생한 원가의 자산화 조건 |
| 19 | **본인 vs 대리인** | 총액(본인) vs 순액(대리인) 인식 |
| 20 | 라이선싱 | 접근권(기간 인식) vs 사용권(일시 인식) |
| 21~26 | 반품권, 보증, 재매입약정, 위탁약정, 미인도청구, 고객의 선택권 등 | 특수 거래 유형별 판단 |

### 합산 토픽 분할 (ADR-16)

초기에는 "통제 이전의 특수 형태"처럼 여러 주제를 하나로 묶어 두었으나, **반품권·보증·재매입약정·위탁약정** 등은 체크리스트와 결론 분기가 완전히 다르다. 이를 개별 토픽으로 분할하여 AI가 정확한 체크리스트를 적용하도록 했다.

## 3. Decision Tree 7개 섹션 구조

각 토픽의 Decision Tree는 **7개 섹션**으로 구성된다.

| # | 섹션 | 역할 |
|---|------|------|
| 1 | **routing** | 이 토픽에 해당하는 키워드 + 판단 목표 |
| 2 | **checklist** | AI가 사용자에게 확인할 질문 목록 |
| 3 | **conclusion_guide** | AI가 선택할 수 있는 결론 분기 |
| 4 | **precedents** | 분기별 참조할 적용사례·질의회신 |
| 5 | **red_flags** | 분기별 감리지적사례 경고 |
| 6 | **calculation_formula** | 해당 시 계산 공식 |
| 7 | **critical_factors** | AI가 특히 주목해야 할 핵심 판단 요소 |

### Decision Tree 작성 방법: AI 교차검증

26개 토픽의 Decision Tree는 다음 과정으로 작성했다:

1. **기준서 원문 분석**: K-IFRS 1115호 본문·적용지침·결론도출근거를 토픽별로 분류
2. **AI 교차검증**: Gemini, Claude, NotebookLM 3개 AI에 동일한 토픽에 대해 체크리스트·결론 분기·핵심 판단 요소를 독립적으로 생성시킨 후, **공통 항목은 채택하고 불일치 항목은 기준서 원문으로 검증**
3. **선례·감리사례 배정**: 질의회신 101건, 감리지적사례 18건을 토픽별·분기별로 수동 매핑
4. **반복 검증**: 실제 질문으로 테스트 → 체크리스트 누락·분기 모호성 발견 시 수정

### 파이프라인 연결: 각 섹션이 어디서 사용되는가

```
사용자 질문 입력
    │
    ▼
[1] routing → tree_matcher가 키워드·임베딩·LLM 추론으로 토픽 매칭
    │
    ▼
[2] checklist + [3] conclusion_guide → AI 프롬프트에 주입
    │                                    "이 체크리스트를 확인하고,
    │                                     이 분기 중 하나를 선택하라"
    ▼
[4] precedents → 문서 ID 추출 → MongoDB에서 원문 직접 조회 (핀포인트 검색)
    │              AI 프롬프트에 "[큐레이션 선례]"로 직접 주입
    ▼
[5] red_flags → 감리사례 ID 추출 → 핀포인트 검색 + AI에 경고 문구 주입
    │
    ▼
[6] calculation_formula → 계산 질문 시 공식 텍스트를 AI에 직접 전달
    │
    ▼
[7] critical_factors → AI에 "특히 주의할 요소" 안내
```

## 4. 대표 토픽 3개 해설

### 토픽 ①: 계약의 식별

가장 기본적인 판단 — "이 거래가 K-IFRS 1115호의 적용 대상인가?"

**체크리스트 (5항목)**:
1. 계약 당사자들이 계약을 승인하고 의무를 확약했는가? (문단 9⑴)
2. 계약에 상업적 실질이 존재하는가? (가공매출/순환거래가 아닌가?) (문단 9⑷)
3. 대가의 회수 가능성이 높은가? (문단 9⑸)
4. 지급 능력이 낮은 고객인 경우, 가격할인 등을 반영한 실제 받을 대가로 평가했는가? (QNA-SSI-38674)
5. 요건 미충족 시 수취한 대가를 부채로 잡았다면, 재화 이전 중단 시점에 수익으로 전환했는가? (EDU-KASB-BC46)

**결론 분기 (4개)**:
| 분기 | 결론 |
|------|------|
| 분기 1 | 5가지 요건 모두 충족 → IFRS 1115호 적용 |
| 분기 2 | 요건 미충족 (가공매출 등) → **수익 인식 불가** |
| 분기 3 | 요건 미충족이나 대가 수취 → **선수금(부채) 처리** |
| 분기 4 | 계약 상대방이 고객 아님 (동업자 등) → 1115호 적용 배제 |

**선례**: IE 사례 1~4, QNA-SSI-36917(구두계약), QNA-SSI-36993(회수 불확실), EDU-KASB-BC46(부채→수익 전환 시점)

**감리 경고**: 밀어내기·가공매출·페이퍼컴퍼니 순환거래 — 감리 지적 빈도 1위

---

### 토픽 ②: 본인 vs 대리인

실무에서 가장 논쟁이 많은 판단 — "총액(본인)으로 인식할 것인가, 순액(대리인)으로 인식할 것인가?"

**체크리스트 (3항목)**:
1. 재화/용역이 고객에게 이전되기 전에 기업이 **통제**하는가? (문단 B35)
2. 유상사급 거래의 경우, 수급사가 원부자재를 임의 처분할 수 없고 재매입약정 실질이 있는가?
3. 소프트웨어 재판매의 경우, 라이선스 키 발급 전에 기업이 통제를 획득했는가?

**결론 분기 (5개)**:
| 분기 | 결론 |
|------|------|
| 분기 1 | 통제함 → **본인 (총액 인식)** |
| 분기 2 | 주선만 함 → **대리인 (순액 인식)** |
| 분기 3 | 유상사급 맞춤형 원부자재 → 실질적 재매입약정 → 대리인(순액) |
| 분기 4 | 본인일 때 → 수수료·운송비는 매출원가에 가산 |
| 분기 5 | 대리인일 때 → 지급액은 순액에서 미차감, 판관비 처리 |

**핵심 판단 요소 (critical_factors)**: ① 가격결정권 보유 여부 ② 재고위험 부담 여부 ③ 이전 전 통제 여부

**감리 경고**: 실질이 대리인임에도 총액으로 매출을 부풀리는 것은 **감리 지적 최다 유형** (FSS-CASE-2024-2409-02, 2024-2505-03, 2022-2311-03)

---

### 토픽 ③: 변동대가

계산 기능이 포함된 토픽 — AI가 기댓값법·최빈값법을 **직접 계산**

**체크리스트 (5항목)**:
1. 대가가 변동할 요인이 있는가? (할인, 리베이트, 위약금, 성과보너스 등)
2. 기댓값과 최빈값 중 어느 방법이 대가를 더 잘 예측하는가?
3. 불확실성 해소 시 수익 환원 가능성이 낮은가? (제약 요건)
4. 환불 예상액이 있는가?
5. 제3자 대신 회수하는 세금(부가세 등)이 있는가?

**결론 분기 (5개)**:
| 분기 | 결론 |
|------|------|
| 분기 1 | 제약 통과 → 기댓값/최빈값으로 추정하여 **수익에 포함** |
| 분기 2 | 제약 미충족 → 불확실성 해소 전까지 **수익에서 제외** |
| 분기 3 | 환불/반품 예상액 → **환불부채로 별도 인식** |
| 분기 4 | 위약금·장려금 → **수익에서 선제적 차감** |
| 분기 5 | 부가세 등 → **거래가격에서 제외** |

**계산 공식 (6_calculation_formula)**: 이 토픽만의 특징 — AI가 사용자의 숫자를 받아 직접 계산

```
[기댓값법] 추정 변동대가 = (결과치 A × A 확률) + (결과치 B × B 확률) + ...
[최빈값법] 추정 변동대가 = 발생 가능성이 가장 높은 단일 금액
[위약금]   인식할 수익 = 총 고정 대가 - 추정 변동대가(기댓값 또는 최빈값)
```

계산 질문이 감지되면 이 공식이 AI에게 전달되고, 산술 정확도 100%인 gpt-4.1-mini가 계산을 수행한다.

## 5. 토픽 매칭: 사용자 질문 → Decision Tree 연결

사용자의 자유형 질문에서 **어떤 토픽의 Decision Tree를 적용할지** 결정하는 것이 토픽 매칭이다. 3가지 독립적인 신호를 결합한다.

### 3중 매칭 구조

```
사용자: "A사가 B사를 통해 소프트웨어를 재판매합니다. 매출은 총액인가요?"
    │
    ├── [신호 1] 키워드 매칭
    │     search_keywords = ["재판매", "총액", "매출"]
    │     → "본인 vs 대리인" 토픽의 trigger_keywords에 "총액" 적중 → +2.0점
    │
    ├── [신호 2] LLM 토픽 추론 (topic_hints)
    │     AI가 추론: "이 질문은 본인 vs 대리인에 해당"
    │     → +3.0점
    │
    └── [신호 3] 임베딩 유사도 (의미적 매칭)
          질문 벡터와 26개 토픽 벡터의 코사인 유사도 계산
          → "본인 vs 대리인" 유사도 0.41 → +4.1점
    │
    ▼
  합산 점수: 9.1점 → 1위 매칭 → Decision Tree 로드
```

### 왜 3중인가?

| 신호 | 강점 | 약점 | 보완 관계 |
|------|------|------|----------|
| 키워드 | 정확한 용어 매칭 | 일상 언어("물건 넘기는 역할만") 미포착 | 임베딩이 보완 |
| LLM 추론 | 문맥 이해 | 3회 실행 시 불일치 가능 | 키워드+임베딩이 보완 |
| 임베딩 | 의미적 유사도 | 절대값이 낮아 단독 판단 어려움 | 키워드와 결합 시 신뢰도 상승 |

## 6. 큐레이션 품질 = AI 답변 품질

큐레이션 개선이 시스템 전체 성능에 미치는 영향을 수치로 확인했다.

### 핵심 개선 사례

| 개선 항목 | Before | After | 개선폭 | ADR |
|----------|--------|-------|--------|-----|
| **핀포인트 파싱 3중 Gap 수정** | 적중률 36% | **76%** | **+40%p** | ADR-22 |
| 감리사례 누락 5건 추가 | 커버 13건 | **18건** | +5건 | ADR-20 |
| 3중 매칭 도입 | 토픽 불일치 빈발 | **안정 매칭** | — | ADR-19, 23 |

### ADR-22: 프로젝트 최대 단일 개선

핀포인트 검색(Decision Tree에 배정된 문서 ID로 MongoDB 직접 조회)에 **3가지 파싱 결함**이 있었다:

1. **소괄호 미파싱**: `(IE사례 1, 4)` → 파서가 `[대괄호]`만 인식
2. **축약 ID 불일치**: `FSS-2022-2311-03` ≠ DB의 `FSS-CASE-2022-2311-03`
3. **섹션 7 미수집**: `critical_factors`에 있는 문단 참조 누락

수정 후 검색 적중률이 36%→**76%**로 상승했다. 이는 AI 모델을 교체하거나 프롬프트를 개선한 것이 아니라, **큐레이션 데이터의 품질을 높인 것만으로** 달성한 성과이다.

> **교훈**: RAG 시스템에서는 AI 모델보다 **데이터 품질**이 성능에 더 큰 영향을 미친다.

## 7. 큐레이션의 한계와 확장성

| 구분 | 내용 |
|------|------|
| **수동 유지보수** | 26개 토픽의 체크리스트·선례·감리사례를 수동으로 관리. 새로운 질의회신·감리사례 발표 시 업데이트 필요 |
| **범위 제한** | K-IFRS 1115호만 커버. 리스(1116호)·금융상품(1109호) 등 타 기준서 교차 질문은 범위 밖 |
| **Gray Area** | 메타버스·DeFi·NFT 등 기준서 제정 시 존재하지 않던 비즈니스 모델은 선례 부족 |

### 확장 방향

- **BIG4 가이드 통합**: 삼일·삼정·안진·한영의 적용 가이드를 토픽별로 추가
- **타 기준서 확장**: 1116호(리스), 1109호(금융상품) 등에 동일한 Decision Tree 패턴 적용
- **반자동 큐레이션**: 새로운 질의회신이 발표되면 임베딩 유사도로 관련 토픽을 자동 추천 → 전문가가 확인 후 배정

---

> **요약**: 이 시스템의 핵심 차별점은 AI 모델이 아니라 **26개 토픽 × 7개 섹션의 Decision Tree 큐레이션**이다.
> 체크리스트로 판단을 구조화하고, 결론 분기로 AI의 자의적 추론을 원천 차단하며, 선례와 감리사례를 직접 주입하여 **검색 엔진이 놓쳐도 핵심 자료가 반드시 전달**되는 구조를 만들었다.
