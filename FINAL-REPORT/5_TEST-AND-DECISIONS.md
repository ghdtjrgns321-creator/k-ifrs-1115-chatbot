### 왜 모델 비교가 필요한가

AI 모델마다 강점이 다릅니다. 어떤 모델은 복잡한 판단을 잘하고, 어떤 모델은 산술 계산을 정확하게 합니다.

이 테스트의 핵심 질문은: **"하나의 AI 모델이 회계 판단과 산술 계산을 동시에 잘할 수 있는가?"**

### 테스트 설계

5개 골든 질문(T1~T5)을 설계하여 난이도와 유형을 고루 분배했습니다.

| ID | 토픽 | 유형 | 난이도 |
|----|------|------|--------|
| T1 | 본인 vs 대리인 — 위탁판매 | 거래 상황 판단 | 최고 |
| T2 | 변동대가 — 이론 | 개념 설명 | 낮음 |
| T3 | 변동대가 — 볼륨디스카운트 | 상황 적용 | 중간 |
| T4 | 계약의 식별 — 가공매출 | 거래 상황 판단 | 중간 |
| T5 | 진행률 계산 — 투입법 | 산술 계산 | 중간 |

T1 질문 1개로 7개 모델을 28회(모델당 4회) 호출하여 초기 선별했습니다.

### 라운드 2: 6개 모델 × 5질문 × 5회 = 150회 — 핵심 발견

전체 5개 질문으로 남은 6개 모델을 150회 호출하여 정밀 비교했습니다. 여기서 **프로젝트의 핵심 발견**이 나옵니다.

| 모델 | 종합 점수 | 회계 추론 | 산술 정답률 | 비용(150회) |
|------|---------|----------|-----------|-----------|
| **Gemini Flash (high)** | **0.81** | **1위** | 20~40% | $0.28 |
| Gemini Flash (medium) | 0.79 | 2위 | 20~40% | $0.26 |
| Gemini Flash (low) | 0.76 | 3위 | 20~40% | $0.14 |
| gpt-4.1-mini | 0.71 | 4위 | **100%** | $0.04 |

**핵심 발견: 회계 추론과 산술 정확도는 단일 모델로 양립 불가**

질문별로 보면 패턴이 명확해집니다:

| 질문 | 최고 모델 | 최저 모델 | 인사이트 |
|------|----------|----------|---------|
| T1 (본인/대리인) | Gemini 0.59 | OpenAI 0.35 | **판단 질문은 Gemini 압도적** |
| T4 (계약 식별) | Gemini 0.97 | OpenAI 0.82 | Gemini 5회 연속 0.97 |
| **T5 (진행률 계산)** | **OpenAI 0.86** | **Gemini 0.78** | **유일하게 OpenAI가 우위** |

T5 진행률 계산에서 Gemini가 반복적으로 틀린 패턴이 있었습니다:

```
[정답] 투입법 진행률 = 실제 투입 원가 ÷ (총 예상 원가 - 낭비 원가) = 500 ÷ 2,400 = 20.83%
[오답] Gemini가 계산한 값 = 500 ÷ 2,500 = 20.00%  ← 분모에서 낭비 원가를 빼지 않음
```

Gemini는 thinking_level(추론 강도)을 어떻게 조절해도 이 산술 오류가 반복되었습니다. 반면 OpenAI(gpt-4.1-mini)는 **100% 정확**했습니다.

> **의사결정 (ADR-2)**: 단일 모델은 불가 → **듀얼트랙 라우팅 채택**
> - 회계 판단 질문 → Gemini Flash (추론 품질 1위)
> - 산술 계산 질문 → gpt-4.1-mini (정확도 100%)
>

### 라운드 3~4: 프로덕션 파이프라인에서 다시 검증

듀얼트랙을 실제 시스템에 적용하자 **새로운 문제 2가지**가 발생했습니다.

**문제 1: 오라우팅 (잘못된 모델에 배정)**

T3(볼륨 디스카운트)은 회계 판단 질문인데, "계산 공식이 있다"는 이유만으로 계산용 모델로 배정되어 점수가 **0.79→0.49**로 급락했습니다.

**해결**: 정규식 기반 분류(계산 공식 존재 여부)를 AI 판단으로 전환. 42/42 = **100% 정확도** 달성.

**문제 2: 산술 정확도 급락 (0.83→0.67)**

계산용 모델에게 "답변 형식을 이렇게 해라"는 지시를 **시스템 설정과 질문 두 곳에 중복**으로 넣었더니, 오히려 산술 정확도가 떨어졌습니다.

**해결**: 포맷 지시를 시스템 설정 1곳에만 유지. 산술 정답률 **0.67→0.83** 회복.

### 라운드별 핵심 지표 변화

| 지표 | 1차 | 2차 | 3차 | 4차 (최종) |
|------|-----|-----|-----|-----------|
| 총 호출 수 | 28 | 150 | 15 | 25 |
| 전체 평균 점수 | — | 0.81 | 0.72 | **0.82** |
| 검색 적중률 | 40% | 32% | 36% | **76%** |
| 포맷 준수율 | — | 100% | 60% | **100%** |
| 환각 방지 | 100% | 100% | 93% | **100%** |

3차에서 점수가 0.72로 **오히려 떨어진 것**이 중요합니다. 개별 기능(모델, 검색, 포맷)을 각각 최적화해도, **시스템으로 통합하면 새로운 문제가 발생**합니다. 4차에서 이 문제들을 해결하여 최종 0.82를 달성했습니다.

## 핵심 의사결정 10가지 (ADR)

프로젝트 전체에서 33건의 의사결정을 기록했습니다. `1_PROJECT-OVERVIEW.md`에서 다루지 않은 10건을 선별하여, **무엇을 결정했고 왜 그랬는지**를 설명합니다.

### ADR-1. LangGraph → PydanticAI + 순수 Python

**상황**: 초기에 LangGraph(AI 워크플로우 관리 프레임워크)를 사용했으나, 5개 단계를 순서대로 실행하는 단순한 구조에 과잉 추상화였습니다.

**문제**:
- 중간 단계에서 문제가 생겼을 때 원인 추적이 어려움
- AI 출력이 정해진 양식을 지키지 않으면 3단계 복구 로직(JSON 파싱 → 정규식 추출 → 기본값) 필요

**결정**: PydanticAI(출력 양식을 코드 수준에서 강제하는 프레임워크) + 순수 Python으로 전환

**효과**: 3단계 복구 로직 완전 제거. AI가 양식을 안 지키면 자동으로 반려하고 재생성하므로, "양식을 지키지 않았을 때의 처리"라는 문제 자체가 사라짐.

### ADR-2. 듀얼트랙 모델 라우팅 — 계산 판단을 AI에게 맡기기

**상황**: 모델 비교(2장)에서 듀얼트랙을 결정했으나, "이 질문이 계산 질문인지" 판별하는 방법이 문제.

**1차 시도 (정규식)**: "계산해", "산정해" 같은 키워드 + 숫자 패턴 + 계산 공식 존재 여부를 정규식으로 확인. **결과: 33% 성공** — AI가 생성한 키워드가 매번 달라져서 조건 충족이 불안정.

**2차 시도 (AI 판단)**: 이미 실행 중인 분석 단계에 "이 질문은 계산이 필요한가?" 판단 필드를 추가. **결과: 42/42 = 100% 성공** — 추가 비용 0원(이미 호출 중인 AI에 필드 1개 추가).

### ADR-7. 전문가 큐레이션 시스템 (Decision Trees)

**결정**: AI에게 자유 추론을 허용하지 않고, 26개 토픽별로 **AI 교차검증으로 작성한 판단 체계** 안에서만 답변하도록 제한.

**Trade-off**: 새로운 사업모델(메타버스, DeFi)이나 타 기준서(1116호 등) 교차 질문에는 답할 수 없습니다. 하지만 이 한계를 받아들인 대신 **환각 방지 100%**를 달성했습니다.

> 이 결정의 상세 내용은 [3_DOMAIN-CURATION.md](3_DOMAIN-CURATION.md)에서 다룹니다.

### ADR-10. 산술 정확도 회복 — "지시를 줄여야 정확도가 오른다"

**상황**: 계산용 AI에게 포맷 지시를 두 곳(시스템 설정 + 질문 앞부분)에 넣었더니, 산술 정확도가 **83%→67%**로 급락.

**원인**: 이 AI 모델은 "스스로 추론하는" 유형이 아니라 "지시를 따르는" 유형입니다. 동일한 지시가 2곳에 있으면, **"답을 계산하는 것"보다 "형식을 맞추는 것"에 더 집중**하게 됩니다.

**해결**: 포맷 지시를 시스템 설정 1곳에만 유지 + 불필요한 개념 설명(~2,000자) 제거.

**효과**: 산술 정답률 **1/3 → 3/3** 회복.

> **교훈**: AI에게 "더 많이, 더 자세히" 지시한다고 결과가 좋아지지 않습니다.

### ADR-17. 멀티턴 대화 안전장치 — "결론 후 불필요 질문 방지"

**문제 1: 결론을 내렸는데 추가 질문을 함**

AI가 "본인에 해당합니다"라고 결론을 내린 후, 다음 턴에서 "로열티율은 어떻게 되나요?"라는 불필요한 추가 질문을 생성. 사용자 입장에서 혼란.

**해결**: `concluded`(결론 도달) 플래그 도입. 한번 결론이 나면 후속 턴에서는 추가 질문 없이 **결론 확인 모드**로만 동작.

**문제 2: 핵심 판단 요소를 확인하지 않고 단정**

"본인 vs 대리인" 판단에서 **통제권·가격결정권·재고위험** 중 어느 것도 확인하지 않았는데, "본인입니다"라고 단정하는 경우 발생.

**해결**: `critical_factors`(핵심 판단 요소) 목록 도입. 이 요소들이 확인되지 않았는데 확정 결론을 내리면 **자동 반려 후 재생성**.

### ADR-22. 핀포인트 파싱 3중 Gap 수정 — 프로젝트 최대 단일 개선 (+40%p)

> 상세 내용은 3장(핵심 개선 Before/After)에서 다뤘습니다. 여기서는 의사결정 맥락만 기록합니다.

**결정 근거**: 검색 적중률 36%는 **AI 모델의 문제가 아니라 데이터 파싱의 문제**였습니다. 3가지 파싱 결함을 동시에 수정한 후 76%로 상승.

**교훈**: AI 시스템의 품질 병목은 AI 모델이 아니라 **데이터 처리의 정확성**에 있을 수 있습니다.

### ADR-23. 토픽 매칭 3중 신호 — "키워드만으로는 불안정"

**문제**: "A사가 B사를 통해 물건을 재판매합니다. A사의 매출액은?" 이 질문의 정답 토픽은 "본인 vs 대리인"인데, 키워드 매칭만으로는 **60%만 성공**.

"재판매"라는 단어가 "본인 vs 대리인" 토픽의 사전 등록 키워드에 없기 때문입니다.

**해결**: 3가지 신호를 결합

| 신호 | 역할 |
|------|------|
| **키워드 매칭** | 정확한 용어 일치 |
| **AI 추론 힌트** | 거래 실질로 관련 토픽 추론 |
| **의미적 유사도** | 질문과 토픽 설명의 문맥 유사도 |

3가지 신호를 합산하면 "재판매" 같은 비정형 표현에도 **토픽 매칭 60%→100%** 안정화.

---

### ADR-27. "더 많이 생각하는 AI"가 더 나쁜 결과를 낸 사건

**문제**: AI의 추론 강도(thinking level)를 최고 수준(high)으로 설정했더니, **같은 질문에 어떤 때는 즉답, 어떤 때는 불필요한 추가 질문**을 하는 비일관적 행동 발생.

**A/B 테스트 결과**:

| 지표 | 추론 강도 high | 추론 강도 medium |
|------|--------------|-----------------|
| 꼬리질문 일관성 | 60% | **100%** |
| 답변 품질 손실 | — | **없음** |

**결정**: 추론 강도를 medium으로 변경.

### ADR-18. 범위 밖 판단 — AI 판단 + 코드 검증 2중 방어

**문제**: "증분차입이자율에 대해 알려주세요"는 K-IFRS 1116호(리스) 질문인데, AI가 "금융요소"(1115호)와 혼동하여 범위 내(IN)로 잘못 판정. **프롬프트만으로는 67% 성공**.

**해결**: AI 판단(프롬프트) + 코드 검증 2중 방어

```
[1단계] AI가 "범위 내(IN)"로 판단
[2단계] 코드가 질문에 타 기준서 전용 용어가 있는지 확인
  → "증분차입이자율" 감지 + 1115호 용어("수행의무", "거래가격") 미발견
  → 강제로 "범위 밖(OUT)"으로 전환
```

**효과**: 범위 밖 감지 **67%→100%**

> **교훈**: 중요한 경계 로직은 AI 판단에만 의존하면 안 됩니다. 코드 레벨의 2차 검증이 필수입니다.

### ADR-32. 실사용 데이터 수집 — 골든 테스트의 사각지대 보완

**문제**: 골든 테스트 53건은 개발자가 설계한 시나리오입니다. **실제 사용자가 어떤 질문을 하고, 어디서 불만족하는지**는 파악할 수 없습니다.

**해결**: 매 응답마다 자동으로 4가지를 채점하고 기록합니다.

| 채점 항목 | 기준 | 예시 |
|----------|------|------|
| 응답 속도 | 24초 이상이면 주의 | "이 응답은 38초 걸림" |
| 인용 커버리지 | 인용 0건이면 주의 | "근거 없이 답변 생성됨" |
| 토픽 매칭 | 매칭률 50% 이하이면 주의 | "2개 신호 중 1개만 일치" |
| 결론 신중성 | 정보 부족 시 단정하면 주의 | "핵심 요소 미확인인데 확정 결론" |

추가로 사용자가 AI 답변 아래 **👍/👎 버튼**을 누를 수 있고, 👎 시 사유를 입력할 수 있습니다.

## 실사용 데이터 수집 시스템

### 골든 테스트 vs 실사용 로그의 상호 보완

| 구분 | 골든 테스트 (53건) | 실사용 데이터 수집 |
|------|-------------------|------------------|
| **설계자** | 개발자 | 실제 사용자 |
| **목적** | 회귀 방지 (기존 품질 유지 확인) | 새로운 문제 발견 |
| **채점** | 사전 정의된 정답 기준 | 규칙 기반 자동 채점 + 사용자 피드백 |
| **개선 흐름** | 테스트 실패 → 코드 수정 | 👎 사례 → 골든셋에 추가 → 회귀 방지 |

### 개선 사이클

```
[골든 테스트]                        [실사용 수집]
  53건 자동 검증                       매 응답 로그 + 자동 채점
      │                                    │
      └── 회귀 방지 ◄──────────────── 👎 사례를 골든셋에 추가
                                            │
                                       사용자 피드백 분석
                                            │
                                       새로운 패턴 발견
```

|----------|----------|----------|------|
| 개념 질문 | 0.98 | 12초 | 최고 품질 |
| 거래 상황 | 0.88 | 26초 | 양호 |
| 범위 밖 | 정상 거절 | 8초 | OUT 판정 정상 |

## 최종 프로덕션 라우팅 규칙

모든 테스트와 의사결정을 반영한 최종 시스템 동작 흐름입니다.

```
사용자 질문 입력
  │
  ▼
[분석 단계] gpt-4.1-mini가 질문을 분석
  ├─ 범위 밖 (OUT) → "K-IFRS 1115호 범위 밖입니다" 안내
  ├─ 계산 필요 (needs_calculation = True)
  │     └─ gpt-4.1-mini로 답변 생성 (산술 100%)
  │        └─ 불필요한 개념 설명 제거, 산술에만 집중
  └─ 회계 판단 필요
        ├─ 후속 턴 (이전 대화 이어감)
        │     └─ Gemini Flash (thinking=low, 빠름)
        │        └─ 분석/검색/재평가 스킵, 캐시된 문서 사용
        └─ 첫 턴
              └─ Gemini Flash (thinking=medium, 추론 최적)
                 └─ 2계층 검색 + Reranker 2차 검증 + Decision Tree 체크리스트
```
