# 검색 엔진 + AI 파이프라인

> 사용자의 질문이 입력되면 **5단계 파이프라인**을 거쳐 답변이 생성됩니다.
> 각 단계는 회계감사 절차에 비유하면 이해하기 쉽습니다.

---

## 1. 5노드 파이프라인: 감사 절차를 코드로 옮기다

질문 하나가 들어오면 5개 노드가 순서대로 실행됩니다.

```
사용자 질문
  │
  ▼
[1] Analyze ─── "감사 계획 수립"
  │  질문의 성격을 파악하고, 어떤 자료를 찾을지 계획
  │
  ▼
[2] Retrieve ── "증거 수집"
  │  2계층 검색으로 관련 문서를 DB에서 가져옴
  │
  ▼
[3] Rerank ──── "증거 평가"
  │  수집된 문서의 관련성을 2차 검증하여 순위 재조정
  │
  ▼
[4] Generate ── "의견 형성"
  │  검증된 문서만을 근거로 답변 생성
  │
  ▼
[5] Format ──── "감사보고서 작성"
     답변에 감리사례 경고·꼬리질문을 추가하여 최종 포맷
```

### 왜 LangGraph를 제거했는가

이 파이프라인은 처음에 **LangGraph**라는 AI 오케스트레이션 프레임워크로 구축했습니다. LangGraph는 복잡한 AI 워크플로우를 시각적 그래프로 관리하는 도구로, 업계에서 널리 사용됩니다. 하지만 이 프로젝트에서는 오히려 **방해 요소**가 되었습니다.

| 항목 | LangGraph 사용 시 (Before) | 순수 Python 전환 후 (After) |
|------|--------------------------|--------------------------|
| **구조** | StateGraph 노드·엣지 정의 + 컴파일 | `state.update(await 노드함수(state))` 1줄 |
| **디버깅** | 중간 상태 추적 어려움 (프레임워크 내부 추상화) | `print(state)` 한 줄로 즉시 확인 |
| **AI 출력 검증** | 3단 폴백 필요 (JSON 파싱 → regex → 기본값) | PydanticAI가 양식 강제 + 자동 재시도 |
| **SSE 스트리밍** | 커스텀 콜백 핸들러 필요 | `async yield` 자연스러운 결합 |
| **파일 수** | graph.py, llm.py, grade.py, hyde_retrieve.py, rewrite.py (5개) | pipeline.py 1개 (200줄) |

**핵심 판단**: LangGraph는 10개 이상의 노드가 조건부로 분기·합류하는 **복잡한 그래프**에 적합합니다. 이 프로젝트의 5개 노드 선형 흐름(analyze → retrieve → rerank → generate → format)에서는 프레임워크의 추상화 계층이 오히려 디버깅 난이도만 높였습니다.

> **비유**: 5개 방을 순서대로 걷는 복도에 교통 신호등 시스템을 설치한 격입니다. 신호등 자체를 관리하는 비용이 복도를 걷는 비용보다 컸습니다.

전환 후 **PydanticAI**를 도입하여 AI 출력 양식을 코드 수준에서 강제합니다. LangGraph의 자유형 텍스트 출력 + 3단 폴백 파싱을 완전히 제거하고, 8개 Agent를 통일된 패턴으로 관리하게 되었습니다.

## 2. [1단계] Analyze — 질문 분석과 라우팅

사용자 질문을 AI(`gpt-4.1-mini`)가 분석하여 **10개 판단 항목**을 채웁니다.

| 판단 항목 | 역할 | 예시 |
|----------|------|------|
| **routing** | K-IFRS 1115 범위 내/외 판단 | "리스 회계처리" → OUT (범위 외) |
| **standalone_query** | 대화 맥락을 반영한 독립형 질문으로 재작성 | "그러면 본인인가요?" → "B사가 소프트웨어를 재판매할 때 본인 vs 대리인 판단" |
| **is_situation** | 구체적 거래 상황 vs 개념 질문 구분 | "변동대가란?" → 개념 / "리베이트 계약인데..." → 거래 상황 |
| **search_keywords** | DB 검색에 사용할 K-IFRS 공식 용어 3~5개 | "포인트 적립" → ["고객충성제도", "고객의 선택권", "독립판매가격"] |
| **needs_calculation** | 계산 결과를 요구하는 질문인지 판단 | "진행률 계산해줘" → True |
| **topic_hints** | 관련 토픽 추론 (최대 3개) | "B사가 재판매..." → ["본인 vs 대리인"] |
| **complexity** | 단순 vs 복잡 (답변 깊이 결정) | 단일 쟁점 → simple / Case 분기 필요 → complex |
| **confusion_point** | 사용자의 혼동 원인 포착 | "세금계산서 발행 = 수익인식이라 생각하시는 것 같습니다" |
| **provided_info** | 사용자가 이미 제시한 판단 요소 | ["재고위험: 우리가 부담", "가격결정권: 없음"] |

### 범위 외 질문 방어 (Scope Guard)

프롬프트만으로는 막을 수 없는 **타 기준서 전용 용어**를 코드 레벨에서 추가 방어합니다.

```
"사용권자산의 리스료를 인식하려면..." (K-IFRS 1116호 리스 관련)
  │
  ├─ "사용권자산", "리스료" → 타 기준서 전용 용어 감지
  ├─ "수행의무", "거래가격" 등 K-IFRS 1115 핵심 용어가 없음
  │
  └─ routing 강제 전환: "OUT" → "이 질문은 K-IFRS 1115호 범위 밖입니다" 안내
```

> **왜 코드로 한번 더 막는가?**  프롬프트에 "1115호만 답하라"고 지시해도, AI는 가끔 인접 기준서 질문에 답하려 합니다. 코드 레벨 방어는 이런 예외를 **100% 차단**합니다.

## 3. [2단계] Retrieve — 2계층 검색

이 시스템의 검색은 **2계층**으로 작동합니다. 빠뜨리면 안 되는 핵심 문서는 1계층이, 넓은 범위의 보충 문서는 2계층이 담당합니다.

### Tier 1: 핀포인트 검색 — "감사조서 직접 조회"

Decision Tree에 **AI 교차검증으로 사전 배정한 문서 ID**로 MongoDB에서 원문을 직접 가져옵니다.

```
"변동대가" 토픽 매칭 → Decision Tree 참조
  │
  ├─ 선례(precedents): IE32~35, QNA-SSI-36917
  ├─ 감리경고(red_flags): FSS-CASE-2020-007
  │
  └─ MongoDB에서 해당 문서 ID로 직접 조회 (검색 엔진 거치지 않음)
```

**장점**: 검색 엔진이 놓칠 수 있는 핵심 문서를 **확실하게** 가져옵니다. 누락 위험 0%.

### Tier 2: 하이브리드 검색 — "추가 증거 수집"

Tier 1이 커버하지 못하는 기준서 본문·적용지침을 보충합니다. 두 가지 검색 방식을 융합합니다.

| 검색 방식 | 원리 | 강점 | 예시 |
|----------|------|------|------|
| **벡터 검색** | 문장의 의미를 숫자 벡터로 변환하여 유사도 계산 | 동의어·표현 차이에도 매칭 | "수익 인식 시점" ↔ "매출 인식 시기" |
| **BM25 키워드** | 정확한 단어 일치 기반 전통 검색 | 문단 번호·조항 정확 매칭 | "문단 B63A", "제35조" |

두 검색 결과를 **RRF(Reciprocal Rank Fusion)**라는 방식으로 통합합니다.

> **RRF란?** 벡터 검색에서 3위, 키워드 검색에서 7위인 문서가 있다면, 양쪽 순위를 종합하여 **최종 순위를 결정**합니다. 한쪽 검색만으로는 놓칠 문서를 다른 쪽이 보완하는 구조입니다.

### 실무 용어 → 공식 용어 자동 확장

감사인은 "쿠폰", "적립금" 같은 실무 용어를 사용하지만, 기준서에는 "고객의 선택권", "계약 부채"로 기술되어 있습니다. 15개 매핑 테이블로 **자동 변환**하여 검색 누락을 방지합니다.

```
사용자: "쿠폰 할인 회계처리"
  │
  └─ "쿠폰" → "고객의 선택권" 추가하여 검색
     → 기준서 문단 B39~B40 정확 매칭
```

### 윈도우 부스팅

같은 섹션의 인접 문단이 함께 검색되면 **서로의 점수를 올려줍니다**. 기준서는 연속된 문단이 하나의 논리를 구성하므로, 한 문단이 관련 있으면 주변 문단도 관련 있을 가능성이 높기 때문입니다.

## 4. [3단계] Rerank — 2차 검증

1차 검색이 가져온 30건의 문서를 **Cohere Cross-encoder**가 재평가합니다.

> **Cross-encoder란?** 일반 검색이 "질문과 문서를 각각 점수화"하는 것과 달리, Cross-encoder는 **질문과 문서를 동시에 읽고** 관련성을 판단합니다. 더 정확하지만 느리므로, 1차 검색으로 후보를 줄인 뒤 2차로 사용합니다.

### 비즈니스 가중치: 문서 유형별 차등 평가

모든 문서의 실무 가치가 동일하지 않습니다. Reranker 점수에 **문서 유형별 가중치**를 적용합니다.

| 문서 유형 | 가중치 | 이유 |
|----------|--------|------|
| 기준서 본문·적용지침 | 1.0 (기준) | 가장 직접적인 근거 |
| 적용사례 (IE) | 0.95 | 참고 가치 높지만 본문보다는 보조 |
| 결론도출근거 (BC) | 0.85 | 제정 의도 파악용, 직접 답변에는 덜 적합 |
| QNA 질문 텍스트 (Q파트) | 0.80 | 질문 텍스트는 검색에 잘 걸리지만 답변 가치 낮음 |
| QNA 답변 텍스트 (A파트) | 1.0 | 실질적 답변 가치 |

### 핀포인트 문서 보호

핵심 설계: Decision Tree에서 가져온 **핀포인트 문서는 Reranker를 통과하되, 탈락해도 경고만** 출력합니다. 전문가가 선별한 문서를 AI 모델이 "무관하다"고 판정하여 제거하는 것을 방지하기 위함입니다.

```
전문가 큐레이션 문서 (Tier 1)  →  Reranker 통과  →  탈락 시 경고 로그
                                                    (관리자가 큐레이션 점검)
일반 검색 문서 (Tier 2)        →  Reranker 통과  →  0.05 미만이면 제거
```

## 5. [4단계] Generate — 듀얼트랙 모델 라우팅

이 시스템은 **단일 AI 모델을 사용하지 않습니다**. 7개 모델을 218회 호출하여 비교한 결과, **회계 추론 품질과 산술 정확도가 단일 모델로 양립 불가**함을 확인했기 때문입니다.

| 모델 | 역할 | 강점 | 약점 |
|------|------|------|------|
| **Gemini Flash** | 일반·상황 질문 | 회계 추론 품질 **1위** (0.81) | 산술 정확도 20~40% |
| **gpt-4.1-mini** | 계산 질문 | 산술 정확도 **100%** | 회계 추론 4위 (0.71) |

### 자동 라우팅

```
질문 입력 → Analyze 단계에서 needs_calculation 판단
  │
  ├─ needs_calculation = True
  │   → gpt-4.1-mini 호출 (산술 집중 모드)
  │   → 적용사례(IE) 문서 제외, topic_knowledge 스킵
  │   → 산술 정확도 100% 유지
  │
  └─ needs_calculation = False
      ├─ 단순 개념 질문 → Gemini Flash (thinking=low, 빠름)
      └─ 복잡한 판단 질문 → Gemini Flash (thinking=medium, 추론 극대화)
```

### 계산 경로 최적화: "산술에 집중하게 하라"

초기 설계에서 계산 질문에도 일반 질문과 동일한 맥락(토픽 지식, 적용사례)을 AI에 제공했습니다. 그 결과 산술 정확도가 **0.845 → 0.67로 급락**했습니다.

원인: 비추론(non-reasoning) 모델인 gpt-4.1-mini는 **맥락이 많아질수록 핵심 계산에 집중하지 못합니다**. 이는 감사팀에 "진행률 계산만 해달라"고 요청했는데 회계 이론서를 같이 주면 오히려 계산 실수가 늘어나는 것과 유사합니다.

**해결**: 계산 경로에서는 **토픽 지식과 적용사례를 의도적으로 제외**하여 AI가 기준서 본문의 계산 공식에만 집중하도록 했습니다. 산술 정확도 **0.67 → 0.83으로 회복**.

### thinking 수준 조절: "깊이 생각하는 것이 항상 좋지는 않다"

Gemini 모델은 `thinking` 옵션으로 **추론 깊이를 조절**할 수 있습니다. 직관적으로는 thinking=high가 가장 좋을 것 같지만, A/B 테스트 결과는 달랐습니다.

| thinking 수준 | 꼬리질문 일관성 | 문제 |
|:---:|:---:|------|
| **high** | 60% | 같은 질문에 어떤 때는 즉답, 어떤 때는 꼬리질문 → 비결정적 |
| **medium** | **100%** | 적정 추론 → 일관된 판단 |

> **교훈**: AI가 "더 많이 생각하면" 불필요한 분기까지 고려하여 오히려 **판단이 흔들립니다**. 회계감사에서도 과잉 분석이 핵심 쟁점을 흐리는 것과 같습니다.

## 6. 멀티턴 체크리스트: 감사 절차서대로 확인

거래 상황 질문에서 AI는 **한 번에 결론을 내리지 않습니다**. 대신 Decision Tree의 체크리스트를 따라 핵심 판단 요소를 하나씩 확인합니다.

```
사용자: "A사가 B사를 통해 소프트웨어를 판매합니다. 본인인가요?"

AI 1턴: "판단에 필요한 핵심 요소를 확인해주세요:
  □ B사가 재화를 통제한 후 고객에게 이전하나요?
  □ B사가 가격결정권을 보유하나요?
  □ B사가 재고위험을 부담하나요?

  현재 정보로는:
  [Case 1] B사가 통제권 보유 → 본인 (총액 인식)
  [Case 2] A사가 통제권 보유 → 대리인 (순액 인식)"

사용자: "B사는 가격 결정권이 없고, 재고는 A사에 반품 가능합니다"

AI 2턴: "[결론] B사는 대리인에 해당할 가능성이 높습니다.
  [근거] 문단 B37~B38에 따르면..."
```

### 양방향 분기 안전장치 (critical_factors)

"본인 vs 대리인"처럼 **결론이 양방향으로 갈릴 수 있는 토픽**에는 특별한 방어가 있습니다.

```
"본인 vs 대리인" 토픽의 critical_factors:
  ├─ 통제 지표 (B37)
  ├─ 가격결정권
  └─ 재고위험

규칙: critical_factors가 모두 확인되기 전에는
      확정 결론(TYPE 2) 금지 → 반드시 조건부 결론(TYPE 1)만 허용
```

> **왜 필요한가?**  AI가 일부 정보만으로 성급하게 "본인입니다"라고 단정하면 위험합니다. 나머지 정보가 추가되었을 때 결론이 뒤집힐 수 있기 때문입니다. 이 안전장치는 **모든 핵심 요소가 확인될 때까지 확정 결론을 코드 수준에서 차단**합니다.

### 동적 시스템 프롬프트: 대화 진행에 따라 진화

매 턴마다 AI에게 전달하는 지시(시스템 프롬프트)가 **실시간으로 갱신**됩니다.

| 주입 항목 | 역할 |
|----------|------|
| 체크리스트 진행 상황 | "3개 항목 중 2개 확인됨" → AI가 나머지 1개만 질문 |
| 이미 확인된 판단 요소 (✅ 표시) | 사용자가 답한 항목에 ✅ → AI가 **재질문하지 않음** |
| critical_factors 상태 | 미확인 항목이 있으면 TYPE 2(확정) 금지 규칙 주입 |
| 결론 확인 모드 | 충분한 정보가 모이면 "이제 결론을 내려도 좋습니다" 허용 |

---

## 7. Fast-path 최적화: 후속 턴 20초 절감

첫 턴에서 이미 문서를 검색·평가했다면, **후속 턴에서 같은 작업을 반복할 필요가 없습니다**.

```
[첫 턴] Analyze(3초) → Retrieve(5초) → Rerank(3초) → Generate(15초) = ~26초

[후속 턴 - 일반] 동일 5단계 반복 = ~26초
[후속 턴 - Fast-path] Analyze, Retrieve, Rerank 스킵 → Generate만 = ~8초
                       + thinking=low로 전환
```

**조건**: 사용자의 후속 메시지가 **이전 꼬리질문에 대한 답변**인 경우에만 적용. 새로운 주제의 질문이면 전체 파이프라인을 다시 실행합니다.

## 8. 토픽 매칭: 3중 검증으로 정확도 확보

질문이 어떤 토픽(변동대가, 본인 vs 대리인 등)에 해당하는지 판단하는 것은 전체 시스템의 정확도를 좌우합니다. **3가지 방법을 결합**하여 매칭합니다.

| 방법 | 원리 | 강점 |
|------|------|------|
| **키워드 매칭** | 질문에 "볼륨 디스카운트" 포함 → "변동대가" 토픽 | 정확한 용어에 강함 |
| **topic_hints** (AI 추론) | Analyze 단계에서 AI가 "이 질문은 변동대가 관련" 추론 | 맥락 파악에 강함 |
| **임베딩 유사도** | 질문 벡터 ↔ 토픽 설명 벡터의 의미 유사도 | 표현이 달라도 매칭 |

세 점수를 합산하여 **상위 3개 토픽**을 선정합니다. 단일 방법으로는 정확도가 부족했지만, 3중 결합으로 **토픽 매칭 정확도를 76%까지** 끌어올렸습니다.

## 9. 프롬프트 설계: 모델 특성별 차별화

같은 지시를 해도 **AI 모델의 유형에 따라 효과가 다릅니다**. 이 시스템은 모델 특성에 맞게 프롬프트를 다르게 설계합니다.

| 모델 유형 | 대표 모델 | 프롬프트 전략 |
|----------|----------|-------------|
| **Reasoning** (추론형) | Gemini Flash | 목표와 제약만 명시. "이 칸을 채워라, 이것만 인용하라" |
| **Non-reasoning** (비추론형) | gpt-4.1-mini | 단계별 사고 절차를 명시. "1단계: 참고문서 확인 → 2단계: 판단 요소 확인 → 3단계: 결론" |

### Citation Locking — 인용 강제

AI가 답변에 사용할 수 있는 근거를 **[참고문서] 섹션에 제공된 문서로만 한정**합니다.

```
프롬프트 규칙:
  - [참고문서]에 포함된 문단만 인용 가능
  - 인용 형식: 반드시 "(문단 XX)" 형태
  - 인용이 비어 있으면 답변 자동 반려 → AI 재생성 (최대 2회)
```

> 이것이 일반 ChatGPT와의 핵심 차이입니다. ChatGPT는 학습 데이터에서 자유롭게 인용하므로 **존재하지 않는 조항을 만들어낼 수 있습니다**. 이 시스템은 DB에서 가져온 원문만 인용 가능하고, 인용이 비어 있으면 답변 자체가 거부됩니다.

### 혼동점 교정

Analyze 단계에서 파악한 `confusion_point`를 AI에게 전달하여, **사용자의 오해를 먼저 교정**한 뒤 답변합니다.

```
confusion_point: "세금계산서 발행 = 수익인식이라 생각하시는 것 같습니다"

AI 답변:
"사용자님이 언급하신 '세금계산서 발행'은 세법상 의무이며,
 K-IFRS 1115호의 수익인식 판단 기준과는 다릅니다.
 수익인식의 핵심 기준은 '수행의무의 이행'입니다. (문단 31~38)"
```

## 10. 에러 방어: 네트워크 장애와 타임아웃

AI 서비스 호출은 **외부 API 의존**이므로 네트워크 장애에 대비합니다.

| 방어 장치 | 동작 |
|----------|------|
| **파이프라인 타임아웃** | 전체 파이프라인에 시간 상한 설정. 초과 시 중단하고 안내 메시지 반환 |
| **노드별 재시도** | 네트워크 오류 시 지수 백오프(2초→4초)로 최대 2회 재시도 |
| **재시도 전 시간 체크** | 남은 시간이 부족하면 재시도하지 않음 (무의미한 대기 방지) |
| **PydanticAI 출력 검증** | 구조화된 출력이 양식에 맞지 않으면 자동 재생성 (최대 2회) |
| **경로 폴백** | clarify 경로 실패 시 → generate 경로로 전환하여 답변 |

## 파이프라인 전체 요약

```
사용자 질문
  │
  ▼
[Analyze] gpt-4.1-mini
  ├─ routing=OUT → "범위 외" 안내 후 종료
  ├─ routing=IN + Scope Guard 통과
  │   ├─ 10개 분석 항목 추출
  │   ├─ is_situation=True → 토픽 매칭 (3중 검증)
  │   └─ needs_calculation 판단
  │
  ▼
[Retrieve] 2계층 검색 (병렬)
  ├─ Tier 1: 핀포인트 (Decision Tree → MongoDB 직접 조회)
  └─ Tier 2: 하이브리드 (벡터 + BM25 + RRF + 윈도우 부스팅)
  │
  ▼
[Rerank] Cohere Cross-encoder
  ├─ 비즈니스 가중치 (본문 > IE > BC > Q파트)
  ├─ 임계값 0.05 미만 제거
  └─ 핀포인트 문서 보호
  │
  ▼
[Generate] 듀얼트랙 라우팅
  ├─ 계산 → gpt-4.1-mini (산술 100%, IE/토픽지식 제외)
  └─ 일반 → Gemini Flash (thinking=medium, 추론 1위)
      ├─ 거래상황: 체크리스트 → 조건부 결론 → 확정 결론
      └─ 개념질문: 직접 답변
  │
  ▼
[Format] 최종 포맷
  ├─ 감리사례 경고 추가
  ├─ 꼬리질문 추가
  └─ 인용 문단 정리 → 좌우 Split View로 표시
```

> **핵심 원칙**: 이 파이프라인의 모든 설계는 **"AI가 자유롭게 답하지 못하게 한다"**는 원칙에서 출발합니다. Analyze가 범위를 제한하고, Retrieve가 근거를 수집하고, Rerank가 노이즈를 제거하고, Generate가 정해진 양식 안에서만 답변합니다. 어느 한 단계라도 실패하면 답변이 생성되지 않습니다.
