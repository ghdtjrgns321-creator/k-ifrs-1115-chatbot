## 왜 이 프로젝트가 필요한가 <sub>[상세: PROJECT-OVERVIEW](FINAL-REPORT/1_PROJECT-OVERVIEW.md)</sub>

### [Problem] 범용 AI의 환각(Hallucination) 현상과 부실 감사 리스크

기존 범용 LLM을 회계 기준서 해석에 그대로 적용할 경우, 근거 없는 답변을 생성하는 **환각 문제**에 노출된다.
이는 감사 실무에서 AI를 **과대신뢰**하는 2종 오류를 유발하여 심각한 리스크로 이어질 수 있다.

> AI에게 자유로운 추론을 허락하지 않고 철저히 통제된 환경에서만 작동하는 **확정적(Deterministic) 아키텍처**가 필요했다.

| 항목 | 일반 ChatGPT | 이 시스템 |
|------|-------------|----------|
| 답변 근거 | 학습 데이터에서 자유 추론 | DB에 저장된 기준서 원문만 사용 |
| 근거 추적 | 불가능 (출처 불명) | 문단 번호 단위로 추적 가능 |
| 환각 방지 | 없음 | 할루시네이션 방어 5-Layer 파이프라인 |
| 결론 구조 | 자유형 텍스트 | Decision Trees 체크리스트 + 분기별 결론 가이드 |
| 정보 부족 시 | 근거 없이 단정 | 조건부 결론 (Case 1/2)으로 분기 제시 |
| 검색 | 없음 | 2계층 하이브리드 검색 + Reranker 2차 검증 |
| 모델 | 단일 모델 | 듀얼 LLM 라우팅 — 질문 유형별 최적 모델 자동 선택 |
| 품질 검증 | 없음 | 53개 골든셋 자동화 회귀 테스트 |

## 핵심 UX — "근거 선행, AI 후행"

이 시스템은 AI에게 바로 질문하는 일반 챗봇과 다르다.
**사용자가 먼저 기준서 원문을 확인하고, AI는 그 근거 위에서만 답변한다.**

### 1. 토픽별 관련 토픽 네비게이션

30개 토픽 중 하나를 선택하면, 연관된 토픽 버튼이 함께 표시된다.
예) "할인액의 배분" → `거래가격 배분`, `변동대가의 배분`, `계약변경`
한 토픽에서 관련 주제로 자연스럽게 이동하며 기준서를 탐색할 수 있다.

### 2. 4개 탭 — 본문·적용사례·질의회신·감리사례

각 토픽에는 사전 큐레이션된 4개 탭이 있다:

| 탭 | 내용 | 출처 |
|----|------|------|
| **본문·BC** | 기준서 원문 + 결론도출근거 | kifrs.com |
| **적용사례** | IE 적용사례 (실무 예시) | kifrs.com |
| **질의회신** | QNA 101건 중 해당 토픽 관련 | kifrs.com |
| **감리사례** | FSS·KICPA 감리지적사례 | 금감원·한공회 |

AI를 호출하기 전에, 관련 근거를 **검색 없이 즉시 열람**할 수 있다.

### 3. 탭별 요약 — 전문을 읽기 전에 핵심을 먼저 파악

각 탭과 개별 항목에 **큐레이션된 요약(summary/desc)**이 제공되어, 전문을 펼치기 전에 내용을 빠르게 훑을 수 있다.

- **본문·BC 탭**: 탭 상단에 해당 토픽의 핵심 개념을 1~2문장으로 요약 (하늘색 박스)
- **질의회신 탭**: 각 QNA 항목마다 한 줄 요약이 표시되어, 어떤 질의인지 펼치지 않아도 파악 가능
- **감리사례 탭**: 각 감리지적사례마다 한 줄 요약으로 쟁점을 미리 확인
- **적용사례 (IE)**: AI 답변 근거 패널에서도 사례별 요약이 함께 표시

> 문서 수십 건을 하나씩 열어볼 필요 없이, **요약만 훑으면서 관련 항목을 골라 상세 열람**하는 흐름을 지원한다.

### 4. 기준서 원문의 관련 조항 연결

기준서 원문을 열람할 때, 해당 문단이 참조하는 **관련 조항이 자동으로 표시**된다.
예) 문단 81 → 관련 조항: `문단 82` + 출처 경로(`본문 > 측정 > 거래가격을 수행의무에 배분함 > 할인액의 배분`)
조항 간 연결을 따라가며 기준서의 논리 구조를 파악할 수 있다.

### 5. Split View — AI 답변 + 근거 문단 동시 확인

```
┌──────────────────────┬──────────────────────┐
│  [좌측] 근거 문서     │  [우측] AI 답변       │
│                      │                      │
│  ▸ 본문·적용지침      │  [결론]              │
│    (AI 인용 문단 볼드) │   Case 1 / Case 2   │
│  ▸ 적용사례 (IE)     │  [논리적 근거]       │
│  ▸ 질의회신 (QNA)    │   문단 B37~B38 기반  │
│  ▸ 감리사례          │  + 유사 감리사례 경고 │
│  ▸ 참고 추가 문서    │  + 꼬리질문          │
└──────────────────────┴──────────────────────┘
```

- **좌측**: AI가 검색해온 근거 문서를 카테고리별로 표시. **AI가 직접 인용한 문단은 볼드 처리**되어 어떤 근거로 결론을 냈는지 즉시 확인 가능
- **우측**: 구조화된 AI 답변 (결론 분기 + 논리적 근거 + 감리사례 경고 + 꼬리질문)
- **참고 추가 문서**: AI 답변에 직접 사용되지 않았지만, Retrieve로 추가 검색된 관련 문서를 "참고보기"로 제공

## 핵심 기술 <sub>[상세: SEARCH-AND-AI](FINAL-REPORT/4_SEARCH-AND-AI.md)</sub>

### 할루시네이션 방어 DATA PIPELINE <sub>[상세: DOMAIN-CURATION](FINAL-REPORT/3_DOMAIN-CURATION.md)</sub>

데이터 적재부터 최종 답변 출력까지 전 과정을 통제하는 **확정적(Deterministic) 아키텍처**를 설계했다.
검증된 **K-IFRS 도메인 지식**만을 AI에 주입하고, 데이터가 **의사결정 트리(Decision Tree)**와 **PydanticAI**를 통과함으로써
AI의 자의적 추론을 원천 차단하고, 정보가 부족한 상황에서도 임의의 판단 대신 안전한 **'조건부 결론(Case 분기)'**을 도출한다.

**5-Layer Data Pipeline:**

1. 검증된 도메인 데이터를 DATABASE에 적재 (본문·적용사례·질의회신·감리사례+KAI교육자료)
2. Pinpoint검색과 Reranker를 통해 가장 정확한 DATA만을 답변 근거로 추출하여 프롬프트에 주입
3. 질의의 복잡도와 유형을 판단하여 최적의 AI 모델로 연결하는 듀얼 LLM 라우팅 적용
4. 토픽별 의사결정 트리(Decision Tree)로 자의적 추론을 차단하고 정해진 분기(Case) 내에서만 결론 도출
5. PydanticAI를 활용하여 '근거'와 '결론'이 완벽히 분리된 구조화된 출력 양식을 코드 수준에서 강제

### Pinpoint검색 + Reranker 교차 검증

| 계층 | 방식 | 특징 |
|------|------|------|
| **Tier 1** | 핀포인트 검색 | Decision Tree에 배정된 문서 ID로 MongoDB 직접 조회. 누락 위험 0% |
| **Tier 2** | 하이브리드 검색 | 벡터 검색(의미 유사도) + BM25(키워드 정확 매칭) + RRF 융합 |
| **Reranker** | 2차 검증 | Cohere Cross-encoder로 재평가. 핀포인트 문서는 보호 |

- **Tier 1 — 핀포인트 검색**: Decision Tree에 사전 배정된 문서 ID로 DB에서 직접 조회. 검색 엔진 미경유 — 누락 위험 0%
- **Tier 2 — 하이브리드 보충 검색**: 벡터 검색 + BM25 키워드 검색을 RRF로 융합. 실무→공식용어 자동 확장 (15개 매핑)
- **Reranker (Cohere Cross-encoder)**: 1차 검색 결과 30건을 질문과 동시에 읽고 관련성을 재평가. 2단계 가중치로 문서의 실무 가치를 차등 반영 (본문·적용지침 1.3 / 감리사례 1.2 / 적용사례·QNA 1.1 / BC·용어·시행일 0.8)

### 듀얼 LLM 라우팅

7개 모델을 218회 호출하여 비교한 결과, **회계 추론과 산술 정확도가 단일 모델로 양립 불가**함을 확인했다.

| 모델 | 역할 | 강점 |
|------|------|------|
| **Gemini Flash** (thinking=medium) | 일반·상황 질문 | 회계 추론 품질 1위 (0.81) |
| **gpt-4.1-mini** | 계산 질문 | 산술 정확도 100% |

### 5노드 파이프라인

```
사용자 질문
  │
  ▼
[Analyze] gpt-4.1-mini — 질문 분석·라우팅 (10개 판단 항목)
  │
  ▼
[Retrieve] 2계층 검색 — 핀포인트 + 하이브리드 (병렬)
  │
  ▼
[Rerank] Cohere Cross-encoder — 비즈니스 가중치 + 핀포인트 보호
  │
  ▼
[Generate] 듀얼 LLM 라우팅 — Gemini(판단) / gpt-4.1-mini(계산)
  │
  ▼
[Format] 감리사례 경고 + 꼬리질문 + 인용 정리
```

## 품질 검증 결과 <sub>[상세: TEST-AND-DECISIONS](FINAL-REPORT/5_TEST-AND-DECISIONS.md)</sub>

### 1단계: 모델 선정 — 7개 모델 218회 A/B 테스트

**회계 추론**과 **산술 정확도**를 단일 모델로 양립할 수 있는지 검증하기 위해, 7개 AI 모델을 4라운드에 걸쳐 비교했다.

| 라운드 | 질문 | 모델 | 반복 | 호출 수 | 목적 |
|--------|------|------|------|---------|------|
| 1차 | T1 (1개) | 7개 | ×4회 | **28회** | 초기 선별 — 2개 모델 탈락 |
| 2차 | T1~T5 (5개) | 6개 | ×5회 | **150회** | 정밀 비교 — 듀얼 LLM 라우팅 결정 |
| 3차 | T1~T5 (5개) | 듀얼트랙 2종 | ×3회 | **15회** | 프로덕션 통합 — 오라우팅·산술 급락 발견 |
| 4차 | T1~T5 (5개) | 듀얼트랙 3종 | ×5회 | **25회** | 최종 개선 — 평균 0.72→0.82 |
|      |      |      |      | **218회** | |

**2차 라운드 결과 (핵심 발견):**

| 모델 | 회계 추론 점수 | 산술 정확도 | 선정 결과 |
|------|-------------|-----------|----------|
| **Gemini Flash** (thinking) | **0.81** (1위) | 20~40% | 일반·상황 질문 전담 |
| gpt-4.1-mini | 0.71 (4위) | **100%** | 계산 질문 전담 |
| Claude 3.5 Sonnet | 0.76 (2위) | 67% | - |
| GPT-4o | 0.73 (3위) | 33% | - |
| 기타 3개 모델 | 0.41~0.69 | 0~33% | - |

**결론**: 회계 추론 1위(Gemini)와 산술 1위(gpt-4.1-mini)를 **듀얼 LLM 라우팅**으로 결합. 질문 유형을 AI가 자동 판별하여 최적 모델로 라우팅한다.

### 2단계: 검색 엔진 테스트 — 6종 스크립트 101회

모델 선정 후, 검색 파이프라인을 개선하고 **6종의 검증 스크립트**로 테스트했다.

| 테스트 | 케이스 | 반복 | 호출 수 | 검증 대상 |
|--------|--------|------|---------|----------|
| 핀포인트 DB 실재 검증 | 26건 | ×1 | **26회** | 큐레이션 문서가 DB에 존재하는지 |
| 26개 케이스 검색 커버리지 | 26건 | ×1 | **26회** | 적중률 76/81 = 94% |
| 핵심 질문 검색 경로 분석 | 5건 | ×1 | **5회** | Pinpoint vs Retriever 분리 분석 |
| calc 라우팅 정확도 | 14건 | ×3 | **42회** | 42/42 = 100% |
| IE bypass E2E | 1건 | ×1 | **1회** | 적용사례 bypass 정상 동작 |
| 단일 질문 디버그 | 1건 | ×1 | **1회** | 검색 경로 추적 |
|                  |      |    | **101회** | |

**개선 결과:**

| 개선 항목 | Before | After | 원인과 해결 |
|----------|--------|-------|-----------|
| 검색 적중률 | 36% | **76%** | 핀포인트 파싱 결함 수정 (데이터 품질) |
| 계산 라우팅 정확도 | 33% | **100%** | 질문 유형별 모델 분리 (듀얼 LLM 라우팅) |
| 검색 속도 | 3.7초 | **0.6초** | DB 개별 조회 100회 → 유형별 일괄 조회 3회 (-84%) |
| 꼬리질문 일관성 | 60% | **100%** | Gemini thinking level 조절 (high→medium) |
| calc 포맷 준수율 | 0% | **100%** | 시스템 프롬프트 중복 제거 |

> **검색 적중률 76%**: Top-1 기준이며, 실제 시스템은 상위 3개를 활용하므로 Top-3 기준 적중률은 100%이다.

## 설계 한계와 향후 발전 방향 <sub>[상세: TROUBLESHOOT](FINAL-REPORT/TROUBLESHOOT.md)</sub>

### 설계 한계 — 자유검색 제한

이 시스템은 AI 챗봇이라기보다 **"규칙 기반 전문 검색 시스템"**에 가깝다.
AI에게 자유로운 추론을 허용하면 더 다양한 질문에 답할 수 있지만,
**근거 없는 답변(환각)이 구조적으로 불가능한 현재의 보장**이 깨진다.

| 대응 가능 | 대응 불가능 |
|----------|------------|
| 체크리스트 Q&A 응답 | 관련된 최신 뉴스 검색해줘 |
| 추가 조건 제시 및 결론 확정 | 코인관련 새로운 거래유형인데 어떻게 처리해야해? |
| 개념 설명 · 계산 질문 | 다른 회사는 어떻게 처리해? |

**유연성은 범용 LLM에 맡기고, 무결성에 집중:**
업데이트를 통해 '자유 질문 모드'를 추가할 수도 있다.
하지만 최신 정보나, 구체적인 정답이 제시되지 않은 새로운 사례 등은 대형 LLM(ChatGPT, Gemini 등)을 활용하는 것이 더 효율적이고 합리적인 선택이다.
본 시스템은 범용 LLM이 보장하지 못하는 '환각이 없는 확정적 근거 제시'를 위해 설계되었다.
대화의 자유도를 잃더라도 엄격한 제약을 유지하는 것이, 해당 프로젝트가 가져야 할 최우선 가치라고 판단했다.

### 향후 발전 방향

- **타 기준서 확장**: K-IFRS 1116호(리스), 1109호(금융상품) 등 별도 Decision Trees 구축
- **BIG4 가이드 임베딩**: 삼일·삼정·안진·한영의 K-IFRS 1115 실무 가이드를 데이터에 추가
- **RAGAS 자동 평가**: Faithfulness·Context Precision 등 자동 정량 평가 프레임워크 구축
